6.02  OVERVIEW OF SUBCHAPTER I

This paragraph provides a brief introduction to the remaining paragraphs in this subchapter.

a.  BDN Commands.   An operator creates and updates BDN records by properly completing a variety of BDN screens.  These screens are accessed through the BDN commands which are summarized in paragraph 6.03.

b.  BDN Processing Stages.  There are five stages of BDN processing.  It is not necessary for a claim to go through each stage in most cases.  In some cases, an operator can go through the required stages in one session at a BDN terminal.  The five stages are:

(1)  Claims Establishment.  Whenever a claim or issue arises, [an operator (or ECAP) creates a pending issue in the BDN system.  See paragraph 6.04.  The issue remains pending until it is resolved by the award or disallowance of benefits.

(2)  Claims Development Processing.  An operator may use BDN to request additional information from claimants, service departments and schools in this stage.  See paragraph 6.05.  This stage is frequently bypassed [as development through BDN] is unnecessary.

(3)  Claims Data Entry Processing.  An operator completes eligibility screens and limited award screens.  [See paragraph 6.06.  This stage is frequently bypassed.]

(4)  Claims Adjudication Processing.  An operator [(or ECAP)] completes eligibility, award, and disallowance screens.  The operator may use all of BDN's award processing capabilities.  See paragraph 6.07.

(5)  Claims Authorization.  [An operator (or ECAP)] approves or disapproves the proposed award or disallowance.  See paragraph 6.08.  Where single signature of awards is permitted, the person who completed the claims adjudication stage may also complete the claims authorization stage.

NOTE:  A claim in BDN is always at one or another of these stages.  Edits sometimes prevent an operator from performing a command which is inappropriate for the current processing stage. 

c.  Claims Processing Scenarios.  Certain combinations of BDN commands and screens are common in practice.  These combinations are described in paragraph 6.09 so that the reader can more readily understand how a typical claim is processed through the various stages.

6.03  BDN COMMANDS

An operator uses the following common commands in processing education claims.  A brief description of what each command does and how an operator uses each command follows.

a.  CEST is the acronym for Claims Establishment.  [CEST actions establish a pending issue for original claims.  A separate CEST is required for each type of original claim (e.g., chapter 30 and 1606).  (BDN security prevents an [operator] from being granted the CEST command when that individual also has both CADJ and CAUT commands.)]

b.  [Reserved for future use.]

c.  CDEV is the acronym for Claims Development.  Using CDEV, an operator causes BDN to send a letter to the claimant or other entity requesting specific types of evidence to support the claim.

d.  CDAT is the acronym for Claims Data Entry.  The CDAT command permits access to BDN screens at the 300 level or below.  Under CDAT, the operator enters the following:

(1)  Data required to determine eligibility and entitlement,

(2)  Data to establish dependency status (ch. 30 only),

(3)  Data to award educational benefits for original or reentrance enrollments for all types of training, and

(4)  Data to amend prior, current, or future enrollments.

e.  CADJ is the acronym for Claims Adjudication.  CADJ permits an operator to perform the same functions as under CDAT but also allows him or her to prepare disallowances and certain special types of awards not allowed by CDAT.

f.  GAD/GAP are the acronyms for Generate and Display/ Generate and Print.  With GAD or GAP, BDN generates and displays or [sends the award to the authorization function] using the data the operator entered on the award and supporting screens.  Unlike the other commands, an operator always enters GAD or GAP in NEXT SCREEN of the award or supporting screens or in SCREEN NUMBER on the Ready screen.

g.  CAUT is the acronym for Claims Authorization.  Using CAUT, an operator authorizes awards or disallowances prepared under CDAT or CADJ or other actions requiring authorization.

h.  "501" is similar to CAUT.  If the operator has single signature authority (CAUT), he or she can enter 501 in the NEXT SCREEN field of an award screen.  After the operator presses the ENTER key, BDN then does the following in one step:  [generates] the award, transfers the issue to the Claims Authorization stage and displays the 501 Authorization screen.

i.  CERT is the acronym for Certification.  Under CERT, an operator certifies pursuit of an IHL (Institution of Higher Learning) course, delivery of an advance payment, attendance in an NCD (Non-College Degree) course, hours worked in OJT (On-the-Job Training) or apprenticeship training, lessons completed in a correspondence course and charges incurred in a flight course.

j.  STOP, SUSP, and RESU are acronyms for stop, suspend and resume, respectively.  Using these commands, an operator prepares stop, suspend, or resume payment actions.

k. DISP is the acronym for DISPose.  With DISP, the operator can dispose (move) the claim to a different processing stage, if necessary.

(3)  [The certification data entered on the 317 screen when an award is being processed may be for that award or for an award already in the master record file.]

c.  Using CERT.  With the CERT command, an operator can enter certification data for any type of training including COD for an existing award in the master record.  BDN uses the data entered to build a transaction and then sends that transaction to Hines BDC (Benefits Delivery Center) for processing.  BDN checks the master record to see if a certification is necessary under CERT.  If a certification is not needed, BDN shows the M21 screen with a message to this effect.  BDN shows the 317 screen with the necessary data from the M/R for completion by the operator if certification data is necessary.  The CERT command does not establish a PIF.

(1)  Ready Screen Data.  Access to the 317 screen through the CERT command is via the Ready screen.  The required entries are:

(a)  CERT and password,

(b)  File number,

(c)  Payee number,

(d)  Benefit [(Ch. 32 or 35]); and

(e)  Claimant's stub name.

(2)  Master Record Search.  When the operator completes the Ready screen and presses the ENTER key, BDN checks the M/R and determines if that record requires certification.

(a)  If certification is not required, the M21 screen is displayed with the message, M/R STATUS INDICATES CERT IS NOT REQD.

(b)  If a matching M/R is not found, the Ready screen is redisplayed with the message, M/R NOT LOCATED FOR REQUEST.

1.  RETURN OF UNDELIVERABLE CHECK SUSPENDED M/R.  This may require an address change.  BDN also shows the following message.

2.  SUSPENDED M/R--OTHER ACTION MAY BE REQD.  This may require other award action. Review the M/R.  BDN does not show this message if the suspense is due to a delinquent certification.

(a)  The benefit indicator, the date, the file and payee number, and the stub name.

(c)  The BEGIN DATE as shown in the M/R as DLC.  The operator may change this date, if necessary.

(a)  TYPE CERT.  The entry in this field must agree with the type of training in the M/R.

(b)  BEGIN DATE.  This date must equal DLC in the M/R.

(c)  CERTIFIED THRU.  This date must be earlier than or the same as the ending date of the award.  Be sure to remember that the date in the M/R is the "none" date and the date entered in CERTIFIED THRU is the "last" day of attendance.  It is possible to certify a period that spans a no-rate period.

6.15  SUPPLEMENTAL AWARD PROCESSING-AWARD LINE ENTRY REQUIREMENTS

a.  General.  For supplemental award processing, there are two methods for entering new or changed award data for chapters 32 and 35.  The two methods are "full input" and "minimal input."  "Full input" means that the operator must enter at least the effective and ending dates and any other changed enrollment data on the first entered award line.  On subsequent lines, the operator must enter only the changed data.  "Minimal input" means that the operator may enter only the necessary data to produce a new award line or to change current or prior award lines.  Both methods for entering new or changed data use a feature called "rolldown."  Under certain conditions, BDN rolls forward data from the displayed M/R award lines or from the operator entered award lines into the next award line field.  Rolldown is available for both chapter 32 and 35 records.

b.  Full Input Requirements

(1)  The operator must use full input when the following conditions exist:

(a)  More than one award line is to be entered.

(b)  Type of training changes category (IHL, NCD, OJT, correspondence).

(c)  The changed line is subject to a rate based on total charges.

(2)  The operator may use full input for any reentrance or amended award.  However, if the operator attempts to use minimal input and BDN requires full input, BDN notifies the operator to this effect.

c.  Minimal Input Requirements

(1)  The minimal input method allows the operator to use existing data in the M/R when entering a reentrance or amended award.  The operator only has to enter the new or changed data and BDN incorporates the operator entered data with the existing data to format an award transaction.

(2)  BDN limits minimal input to changing one award line when the type of training does not change from one category to another from the effective date of the change to the final no-pay date.

(3)  Generally, an operator can enter the following types of changes using minimal input:

(a)  Entering the effective and no-pay dates of a reentrance award when the rest of the award line is the same as the latest line in the M/R;

(b)  Changing an effective date or a no-pay date or both; or

(c)  Changing training time.

(4)  BDN recognizes minimal input by the reason code entered by the operator.  BDN keys each type of minimal input change to a specific reason code.  Some reason codes tell BDN to change only the data in that line and to leave any subsequent M/R award lines unchanged.  Other reason codes tell BDN to change training time in the operator entered line and all subsequent lines.  (See paragraph 6.16 for information concerning minimal input codes.)

d.  Rolldown Requirements

(1)  Rolldown is one of the features of BDN that allows minimal input awards and assists in some full input awards.  With rolldown, BDN rolls forward data in a particular field from the M/R award lines or the operator entered award lines.  Rolldown occurs in all minimal input awards and under most conditions with full input awards.

(2)  Under minimal input, BDN rolls down any data not changed by the operator.

(3)  Under full input, BDN rolls data down under the following conditions:

(a)  For reentrance awards, BDN does rolldown from the latest M/R award line.

(b)  For amended awards with the first entered line having an effective date, the rolldown decision is based on that effective date and possibly the first no-pay date, as follows:

1.  If the entered effective date falls within an award period, rolldown is done from that M/R award line; or

2.  If the entered effective date does not fall within an award period, then the earliest entered no-pay date affects the rolldown decisions as follows:

a.  If the earliest entered no-pay date extends past the effective date of the "next" M/R award period, the rolldown is from that ("next") award line; or

b.  If the earliest entered no-pay date does not extend past the effective date of the next M/R award period, rolldown is from the "previous" M/R award line.  However, if the entered effective date and the earliest entered no-pay date are both prior to the earliest M/R effective date, BDN does not perform rolldown.

(4)  For amended awards with the first entered line having a no-pay date only, BDN bases a rolldown decision, for rolling data into the first entered line which has an effective date, on where the entered no-pay date (on the first line) falls within the M/R award lines, as follows:

(a)  If the entered no-pay date equals the effective date of an M/R award line, BDN performs rolldown from the previous M/R award line.  However, if the entered no-pay date equals the effective date of the first M/R award line, BDN does not perform rolldown; or

(b)  If the no-pay date falls anywhere else within an M/R award line, then BDN does rolldown from that M/R award line.

6.17  GAD/GAP/501 PROCESSING

a.  Overview.  [GAD/GAP/501 processing involves checking proposed awards with complex edits before BDN builds a transaction.  BDN divides the GAD/GAP/501 process into several broad areas of processing.]

(1)  Preliminary GAD/GAP/501 Processing.  BDN edits insure that an operator takes award action using the proper command and using the proper combination of screens.  BDN edits minimal input actions for appropriateness.

(a)  During preliminary processing, BDN builds a "full input" award from an award for which the operator entered "minimal input" data (chs. 32/35 only).  Once BDN generates a full input award from minimal input data, BDN regards the award as a "full input" award as if the operator had entered the data.  If the operator finds it necessary to make changes on the award or to reenter GAD/GAP 501, BDN treats the award as a full input award.  This may require the operator to enter all the changes a second time.

(b)  If the operator prepares the award using CDAT, BDN checks to see whether the award action meets certain criteria that may require CADJ authority.  If BDN locates the following types of awards prepared under CDAT, BDN shows the Ready screen with the appropriate message:

1.  Special Restorative Training: SRT CASE - USE CADJ COMMAND ("I" entered in Type Training).

2.  Independent Study: INDEPENDENT STUDY - USE CADJ COMMAND.

3.  Incarcerated Claimant: INCARCERATED CASE - USE CADJ COMMAND.

(c)  For chapter 35, if BDN locates multiple issues for the file and payee numbers with one of those issues already GAP'd or authorized, BDN shows the Ready screen with the message, PREVIOUS AWARD PENDING AUTHORIZATION or PREV AWD PENDING-RESUBMIT AFTER CYCLE.

(2)  Eligibility and Entitlement Processing.  During this processing, BDN performs the following edits:

(a)  If the operator does not use the 310 or 311 screen, BDN verifies the M/R eligibility fields for appropriateness and consistency.

(b)  BDN insures that the entire award is within the period of eligibility.

(c)  BDN computes the entitlement charged for the new award.  If the computed entitlement exceeds remaining entitlement, BDN shows information messages that include the exhaust-entitlement date, depending on the length of the entitlement extension, if one is appropriate.

(d)  BDN generates an entitlement code.

(3)  Rate Generation.  GAD/GAP processing generates rates or verifies  rates entered by the operator.  BDN also generates legislative increases and OJT step rates (ch. 35), when applicable.

(4)  Certification Processing.  This process determines whether certification is necessary (includes both certification of delivery and certification of attendance).  If the operator completed a 317 screen, BDN determines whether the certification entered is for the existing M/R or the new award.

(5)  Wrap-up Processing.  This process ties all loose ends together.  BDN generates the transaction code and, if appropriate, shows informational messages to the operator.

(6)  GAD/GAP/501 Processing.  After the operator completes all the above processing, BDN shows the award if the action was GAD'd.  BDN moves the pending issue to Authorization if the operator GAP'd the award, and updates the PIF to show a [generated award action.  If the operator enters 501 in NEXT SCREEN (SCREEN NUMBER of the Ready screen), BDN performs the same edits as for GAP, generates]  the award, and displays the 501 screen for authorization.  (See par. 6.08 for CAUT procedures.)

b.  GAD Procedures.  The operator uses GAD to check what the award looks like before BDN prints the award.  BDN performs the GAD/GAP/501 edits as described above.

(1)  When the operator enters all the necessary data on all applicable screens, he/she enters GAD in one of the following:

(a)  NEXT SCREEN on any 200, 300, or 400 Selection or processing screen, or

(b)  SCREEN NUMBER on the Ready screen.

(2)  BDN shows the award on whichever award screen the operator used to enter the award data.  When using minimal input, GAD is automatic, regardless of NEXT SCREEN entry, ensuring generation of the appropriate award.

(3)  If any of the information shown is not correct, the operator can make the correction on the screen shown or on any other screen.  If the operator uses minimal input, BDN requires correction of all incorrect entries.

c.  GAP/501 Procedures.  GAP is the final action in preparing an award or a disallowance before the authorization stage.  Once the action is GAP'd, the claim must go to Authorization to be authorized or not authorized if any of the action is incorrect.  (See par. 6.08 for authorization procedures.)  When the award or disallowance is GAP'd, BDN performs the edits described above.  Once the authorizer approves the claim, BDN sends that transaction to Hines.

(1)  The operator enters GAP in the same manner as GAD above.  He/she may enter GAP on the GAD screen shown if the award is correct.

(2)  After GAP'ing, the operator [verifies all the entries,] and forwards the claim to the authorizer.  GAP automatically disposes the pending issue to the Authorization stage.  (See par. 6.08 for authorization procedures.)

(3)  If the operator has CAUT (single signature) authority, he/she may enter "501" in NEXT SCREEN instead of GAP.  BDN requires authorization of every award.  Single signature authority simply allows one individual to perform both the award and the authorization functions.  BDN performs the same edits for 501 as GAP, but immediately shows the 501 Authorization screen for action.  (See par. 6.08 for completing the 501 screen.)  If the operator does not have CAUT authority or the award requires two signatures, the operator should enter GAP and proceed as in subparagraph (2) above.

(4)  BDN requires two signatures in the following cases: (See Part 3: Chapter 1 - Administrative Issues  for list of all education actions requiring two signatures.)

(a)  Original awards,

(b)  Independent study,

(c)  Special restorative training,

(d)  Awards for cases with A/R's (accounts receivable),

(e)  Awards over 5 years in the past,

f)  Awards with excessive extensions of entitlement,

(g)  Chapter 35 and OJT/apprenticeship awards that extend beyond the date entitlement is exhausted,

(h)  Reentrance awards with effective dates greater than 1 year prior to the date of claim,

(i)  Incarcerated claimant cases, and

(j)  Requests for special payments (HELP).

(5)  After GAP'ing (501), if any of the award data is incorrect, the award/disallowance must go through CAUT processing to rescind the authorization, then back through CDAT or CADJ to correct the errors and then re-GAP'd.  A second person must authorize (CAUT) the award/disallowance if the award is re-GAP'd by the same person.

d.  [Reserved.]

Review to see if the delimiting date has been correctly generated, including allowable extensions.  If incorrect, refer to BDN 311 screen as described in Part 7: Chapter 2 - BDN Processing.  If correct, enter the claimant's delimiting date and the proper reason code in the NO-PAY DATE/RSN fields on the award screen.

BDN does not require certification if the no-pay date is earlier than the M/R DLC.

Delete the 317 screen.

(3)  AWARD PRIOR TO INITIAL ENTRY DATE

BDN shows this message on the 312/412 screen if the effective date is earlier than the M/R initial entry date and effective reason is not 75.

Correct the effective date or change the reason code to 75.

(4)  BASIC ELIGIBILITY NOT ESTABLISHED

Establish eligibility by entering either the 310 or 311 screen on an original award.

(5)  CERT BEGIN DATE NOT EQUAL TO AWARD BEGIN DATE

On original awards, the BEGIN DATE entered on the 317 screen must be the same as the effective date of the award.

On amended type training A or B awards, the certification BEGIN DATE must equal the first award effective date.

Correct the BEGIN DATE on the 317 screen.

(6)  CERT BEGIN DATE NOT EQUAL DLC: DLC = (MM-DD-YY)

Two conditions produce this:

(a)  For a reentrance award, the BEGIN DATE on the 317 screen must be the same as the M/R DLC.

(b)  For amended awards, these two dates must be the same if type training is not A or B.

Correct the BEGIN DATE on the 317 screen.

(7)  CERT BEGIN DATE SHOULD EQUAL MM-DD-YY

On an amended award, the BEGIN DATE on the 317 screen should be the same as an award effective date.  MM-DD-YY is the same as the first award effective date after the M/R DLC.

Correct the BEGIN DATE on the 317 screen.

(8)  CERT NOT NEEDED

BDN does not require a certification if the last award no-pay date is earlier than or the same as the M/R DLC.  The message appears on the 317 screen.

Delete the 317 screen.

(9)  CERT THRU DATE BEYOND M/R AWARD END DATE

The CERT THRU date on the 317 screen must be earlier (by at least 1 day) than the last M/R no-pay date.

(10)  CERT THRU DATE BEYOND AWARD END DATE

The CERT THRU date must be earlier (by at least 1 day) than the last award no-pay date if type training is A or B and training time is less than 6.

(11)  CERT THRU DATE HITS NO RATE PERIOD

(a) On an original award, if the 312 or 412 screen is entered, an operator must enter a CERT THRU date earlier (by at least 1 day) than the no-pay date.

(b) On an amended award, the CERT THRU date must be earlier (by at least 1 day) than the first no-pay date, but later than the 317 BEGIN DATE if Type Certification is NCD or APP/OJT.

(c) On a reentrance award, the CERT THRU date must be earlier (by at least 1 day) than the first award no-pay date if Type Certification is NCD/FARM or APP/OJT.

(d) For all supplemental awards, if Type Certification is IHL with training time less than 6, the CERT THRU date:

1.  Cannot fall in a no-rate period and

2.  Cannot be the same as or be later than the last no-pay date, but

3.  May cross a no-rate period.

Correct the CERT THRU date on the 317 screen.

(12)  CERT THRU DATE NOT BEFORE DELIMIT DATE

BDN shows this message if the award is correspondence and the CERT THRU date exceeds the delimiting date.

Correct the CERT THRU date on the 317 screen.

(13)  CERT TYPE INDICATED INVALID

If the M/R is type A or E and training time is other than 6, CORRES Type Certification may not be "Y."

Correct the 317 screen.

(14)  CHARGES MUST BE ENTERED

An entry in CHARGES is required in [chapter 35] cases when a no-pay date is present if the payee is in service or TNG TME is 0 or 1.

(15)  CHECK FOR 6 MONTH STEP

On an APP/OJT award with type training G, the total number of months paid cannot exceed 24.  (Step 4 cannot contain more than 180 days.)

(16)  DELIMITING DATE MISSING, USE 311 SCREEN TO CORRECT

An operator must calculate a delimiting date for a chapter 35 claim by entering one of the three date fields on the 311 screen (Date of Death, P&T Disability or POW/MIA).

If an operator enters an award effective date the same as an M/R no-pay date with an effective reason of 60 or a no-pay date only, he/she cannot enter data in the following fields: RATE, HOURS TYPE, HOURS NUMBER, DAYS PER WEEK, TRNG TIME, ENT CHARGE, TYPE TRNG and CHARGES.

The EOD must always be earlier than the RAD unless the claimant is in service.

The entered rate cannot exceed the maximum table rate, and must be within $1.00 plus or minus.

In an SRT case, the entered rate must be less than the table rate.

As a result of a legislative change or an OJT step decrease, the total number of generated effective dates exceeds 12 or the number of generated effective plus no-pay dates exceeds 16.  Applies to chapter 35 cases only.

Prepare the award in two or more parts, allowing a cycle to process between each part.

(29)  EXCESSIVE INDEPENDENT STUDY RATE

The rate entered exceeds the calculated monthly rate or table rate.

(30)  INITIAL ENTRY SEG DOES NOT EXIST-NOTIFY STO

Both eligibility and award data are entered but no BIRLS record exists.

(31)  INDEPENDENT STUDY ASSUMED

An operator entered CHARGES, TYP TNG is A or B, and the claimant is not a serviceman.

An EOD date must be present in the M/R.

(33)  INVALID PRIOR VA TRNG AMT, USE 310 SCREEN TO CORRECT

If present, the M/R PRIOR VA TRNG entry must be numeric.  The first two positions must be 00 through 99 and the last two, 00 through 29.

(34)  INVALID REQUEST-AWARD AND DISALLOWANCE

The M/R is a type A (running) award and a disallowance has been entered.

On a correspondence award, the number of LSNS AUTHORIZED on the 313 screen must be greater than or the same as the number entered in CORRESPONDENCE LESSONS SERVICED on the 317 screen.

An operator must enter a number of lessons in CORRESPONDENCE LESSONS SERVICED greater than the number of lessons previously serviced and certified.

The operator entered "Y" in either CLEAR CONTROL on the award screen or CERT OF DELV on the 317 screen and BDN does not require a certification of delivery.  BDN shows this message on the 312/412 screen.

(38)  M/R NEEDS CERT OF DELV OR CLEAR CONTROL

The entry in either CERT OF DELV or CLEAR CONTROL is not "Y".  An operator must make an entry in one of these fields if type training in either the PIF or M/R is not A or B.  This message appears on the 317 screen.

(39)  M/R REQUIRES CERTIFICATION

On a reentrance award, an operator must enter a certification if type training is not A or B.  The M/R has a period of training needing certification.  If the folder does not contain this certification, the operator must obtain it.

BDN cannot calculate or derive a rate from the rate table with operator provided information.

BDN does not store generated award lines if the total number of award lines is greater than 12 or the total number of dates (effective and no-pay) is more than 16.

An operator has previously certified the maximum number of lessons for a correspondence course.  (Charges for flight courses are no longer used.)

If the DISAB EXT date is present on the 310 screen, the award effective date cannot be earlier than either the latest RAD plus 10 years or the DISAB EXT date.

A minimal input award cannot be generated for a case where the monthly rate is more than the maximum rate found in the tables.

(45)  PRIOR G OR K TYPE TRNG.  FULL INPUT AWD REQD

If APP/OJT type training is present in the current award line, BDN cannot generate a minimal input award.

(46)  PRIOR VA TRAINING CODE INVALID, USE 310 SCREEN TO CORRECT

If present in the M/R, the prior VA training code must be numeric in the range 0 through 9.  (See ch. 9, par. 9.13.)

(47)  RATE EXCEEDS STATUTORY LIMIT

The rate entered by the operator exceeds the maximum monetary rate in the rate table.

(48)  RATE NOT SUPPORTED BY CHARGES

In a chapter 35 case, if an operator enters a RATE, the rate must be within $1.00 (plus or minus) of any "computed monthly rate" but not greater than the maximum monthly rate found in the table.

All award lines where CHARGES are a factor will have values in RATE used to compute CHARGES.  An error is present if CHARGES entered on the award line are greater than the computed charges.

An operator has entered a minimal input amended award and the claimant is in service.

(50)  TRNG TIME CHANGE IN NO-RATE PERIOD or TRNG TIME CHANGE HITS NO-RATE PERIOD

The EFFECTIVE RSN is 70 or 74 and the effective date entered falls between the last no-pay date and the following effective date.

(51)  TWO CERTS ENTERED AND INCONSISTENT

The operator has completed a 317 screen twice for the same case.

(52)  TYPE TRNG AND TYPE CERT INCONSISTENT

If the operator uses a 312/412 screen and TYP TNG is A or B, he/she must enter "Y" in IHL on the 317 screen.  If TYPE TNG is C, D, [  ] F, H, I or L, the operator must enter "Y" in NCD/FARM.  If TYPE TNG is G or K, the operator must enter "Y" in APP/OJT.  If the operator entered the award on the 313 screen, he/she must enter "Y" in CORRES.

Review the BIRLS record and make any necessary corrections.

Review extension of entitlement.

(12)  POTENTIAL SCHOOL LIABILITY EXISTS

The date in the CONTINUE CONTROL SUSPENSE DATE field on the 501 screen is not at least 30 days in the future.

(16)  SRT AWARD WILL EXCEED 45 MONTHS ENTITLEMENT-HIT ENTER TO CONTINUE

On an original or reentrance award, the entitlement used through the end of the award period exceeds the result of prior VA training minus 45 months.

6.19  406 STOP-SUSPEND-RESUME SCREEN

a.  General.  This screen is available for chapters 32 and 35 only.  [See Part 7: Chapter 2 - BDN Processing, paragraph 2.14 for chapter 35 stop, suspend and resume processing.]  An operator uses the 406 Stop-Suspend-Resume screen for stop, suspend or resume actions only and accesses this screen using the commands STOP, SUSP or RESU.  BDN precludes accessing this screen under CADJ.  The 406 screen is not listed on the 400 Selection screen.  Each of the three actions requires authorization via the 501 Authorization screen.  (See Part 4: Chapter 11 - Enrollment Adjustments for stop, suspend, and resume procedures.)

b.  Access.  An operator accesses the 406 screen via the commands STOP, SUSP and RESU from the Ready screen.

(1)  The Ready screen entries are:

(a)  The command-STOP/SUSP/RESU, as appropriate,

(b)  The operator's password,

(c)  The file number,

(d)  The payee number,

(e)  The end product,

f)  The benefit (Ch. 32/Ch. 35),

(g)  The stub name of the payee.

(2)  When an operator presses the Enter key from the Ready screen, BDN first checks the PIF for an existing pending issue.  If a pending issue exists (under CDAT/CADJ), BDN shows the 406 screen.  If more than one pending issue exists, BDN shows the S02 screen for selection of the desired pending issue.

(3)  If BDN does not locate a pending issue, BDN searches for a master record.

(a)  If the M/R is other than a type A, D, or E record, BDN shows the Ready screen with the message INVALID REQUEST-NO RUNNING AWARD.

(b)  If the M/R is a type D record, BDN shows the Ready screen with the message INVALID REQUEST-M/R IS ACCOUNTS RECEIVABLE.

(c)  If the command is RESU and the M/R is a running award (type A record), BDN shows the Ready screen with the message INVALID REQUEST-M/R IN RUNNING AWARD STATUS.

(d)  If the command is SUSP and the M/R is already suspended (type A record), BDN shows the Ready screen with the message INVALID REQUEST-M/R IS SUSPENDED.

(e)  If the command is RESU and the master record suspense reason code is 31 (Suspended pending recognition of fiduciary), BDN shows the Ready screen with the message INVALID REQUEST-PENDING APPT OF FIDUCIARY.

(4)  When BDN finds an M/R, BDN shows the 101 screen.  An operator should complete the 101 screen as described in paragraph 6.04f and leave the NEXT SCREEN blank.  BDN automatically shows the 406 screen.

NOTE:  If, for any reason, the operator wishes to stop the action either before or after completing any of the screens in the process, but before he/she presses the Enter key, the operator may enter END in NEXT SCREEN. This action will terminate the action and the pending issue remains on the PIF.  Use PCAN or PCLR to remove the pending issue from the PIF.
