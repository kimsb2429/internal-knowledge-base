Topic 1: Searching for a beneficiary in BM

Introduction

This topic discusses searching for beneficiaries in BM.

Change Date

June 24, 2025

II.3.A.1.a

Searching for a beneficiary

The Search beneficiaries screen is the first screen a user is directed to after logging into BM via https://iam.education.va.gov/. 

The user starts by searching the claimant database for existing beneficiary profiles in BM. If a beneficiary is found in this search, they already exist in BM and the user does not need to create a new profile.

Figure: Claim creation workflow

The user can begin the search be entering any of the following in the corresponding fields on the screen.

  1. Claimant ID:
    1. When searching by a social security number (SSN), the user enters nine (9) numeric digits without dashes.
    2. When searching for an alternate identifier (such as a “C number”), the user enters eight (8) numeric digits without spaces or dashes.

Note: “Wild card” searches are not available in BM.

Note: If the beneficiary does not have an SSN or alternate ID, see Creating a Profile Biographical Information for processing instructions.

  1. First name, Middle name, or Last name:
    1. When searching by any name, the user enters a minimum of one (1) character and a maximum of 40. Searches cannot use first name, middle name, or last name only.
  2. Date of birth:
    1. When searching by date of birth, the user enters the date in MM/DD/YYYY format.  Searches cannot use date of birth only.

Note: For chapter 35 beneficiaries, the user searches using the beneficiary information, not that of the Veteran.

The user selects the Search button to search the claimant database of all beneficiaries established in the DGI Managed Service using the information entered.  The user may select the Clear search button to clear all information from the search fields.

Figure: Search beneficiaries screen

II.3.A.1.b

Search results

After selecting the Search button, the search fields will be grayed out and two new sections appear.

  1. Didn’t find who you were looking for?
    1. This section allows the user to either:
      1. Search the VA/DoD Identity Repository (VADIR) using the Search VADIR button, or
      2. Try a new search using the Restart search button.
  2. Search results allow the user to access the beneficiary profile using the Go to Profile button.

Figure: Search results example

An error message will display if the search has more than 100 results requesting the user to refine the search further.

A message will be displayed if no results are returned with BM.

II.3.A.1.c

Automatic system checks

Two automatic system checks are built into BM searches that can change the way the user sees the search results.

  1. The Benefits Delivery Network (BDN) check, also referred to as the Traffic Marshal, is designed to prevent a user from creating a claim in BM for a beneficiary who already exists in BDN. 

If the beneficiary exists in BDN at the time of beneficiary profile creation, a message displays directing the user to BDN.  The user cannot access the beneficiary via BM and must process the claim in BDN.

Note: All claims will be processed in BM after BDN is fully retired in Summer 2025.  In the interim, only claims with no BDN record will be processed in BM.  See Procedural Advisory: Benefits Manager Rollout Processing Controls for details. Once BDN data migration has completed, Traffic Marshal functionality will be discontinued to allow for BM claims processing.

  1. The BM sensitivity check prevents users from accessing a beneficiary’s record without the required sensitivity level permissions.  A message displays notifying the user that they do not have permission to view the file and provides the sensitivity level for the beneficiary.

Figure: Sensitivity check example

II.3.A.1.d

Searching VADIR

If the desired beneficiary does not appear in the claimant search, the user selects the Search VADIR button to search the VADIR database.  If more than 50 results are returned, the user will be asked to provide additional information. 

If the VADIR search returns any results, the user can create a profile for the beneficiary using the Create claimant button.  The user can also start a new search using the Restart search button. 

Do NOT create a beneficiary profile in BM without first completing an MPI search.  If the MPI search returns no appropriate record, the user must restart the search and then select Create claimant on the VADIR search.

Figure: VADIR search results example

II.3.A.1.e

Searching MPI

If the desired beneficiary still does not appear, the user selects the Search MPI button to search the MPI database.  If more than 10 results are returned, the user will be asked to provide additional information.

Note: The MPI search requires the beneficiary’s last name.  If last name was not previously entered, an entry in the Last name field is needed.

If the MPI search returns any results, the user can create a profile for the beneficiary using the Create claimant button.

Figure: MPI search results example

If no beneficiaries are found using any of the searches, the user can add the beneficiary using the Create a claimant button.

If any results display for a given Claimant ID, the user will not be able to create a new claimant for that ID.  This is to prevent duplicate files in BM.

Topic 2: Beneficiary profile creation

Introduction

 This section discusses creating a beneficiary profile in BM.

Change Date June 24, 2025

II.3.A.2.a

General

 If no existing profile is found in the DGI Managed Service for the searched for beneficiary, the user will need to create a profile before any claim can be created for adjudication.

II.3.A.2.b

Creating a profile

If a corresponding record is not found using the MPI search, but is found using the VADIR search, the user can use the matching Create claimant button from the VADIR search results to open the Create claimant window.  Users also can use the Cancel or X button to close the Create claimant window without taking any actions.

The Create claimant window displays the beneficiary’s biographical and contact information pulled from MPI or VADIR depending on the scenario.  This information cannot be edited in this window but can be adjusted as necessary in the Beneficiary profile screen.

If no matching MPI or VADIR results are found, the user selects the Create a claimant button to open the Create a claimant window to enter the necessary biographical and contact information.  The user can also start a new search using the Restart search button.

Figure: Create a claimant window

After all information has been entered, user selects the Create a claimant button to create the beneficiary profile.  The user selects the Create button to create the profile.  Users can also use the Cancel or X button to close the Create a profile window without taking any actions.

Users must always search MPI before creating any original claim.

II.3.A.2.c

Creating a profile – Biographical information

If no matching MPI or VADIR results were found, the user enters the following biographical information in the Beneficiary biographical information section of the Create a profile window to create a new beneficiary profile.

  1. First name:  This is a required field. 
    1. Must be between one (1) and 40 characters.
    2. Allows the following characters:
      1. All letters (capital and lowercase),
      2. Space,
      3. Hyphen, and
      4. Apostrophe.
    3. When only one (1) character is entered, it only accepts uppercase A through Z.
  2. Middle name:  This is not a required field but must be entered if provided.
    1. If entered, must be between one (1) and 40 characters.
    2. Allows the following characters:
      1. All letters (capital and lowercase),
      2. Space,
      3. Hyphen, and
      4. Apostrophe.
    3. When only one (1) character is entered, it only accepts uppercase A through Z.
  3. Last name:  This is a required field.
    1. Must be between one (1) and 40 characters.
    2. Allows the following characters:
      1. All letters (capital and lowercase),
      2. Space,
      3. Hyphen, and
      4. Apostrophe.
    3. When only one (1) character is entered, it only accepts uppercase A through Z.
  4. Suffix:  This is not a required field but must be entered if provided.
    1. If entered, must be between one (1) and 40 characters.
    2. Allows the following characters:
      1. All letters (capital and lowercase),
      2. Numbers, and
      3. Period.
  5. Date of birth:  This is a required field.
    1. Must be in MM/DD/YYYY format.
    2. The user can either type in the date or select the appropriate date from the calendar icon.
  6. Claimant ID:  This is a required field.
    1. Must be either eight (8) or nine (9) characters.
    2. Allows the following characters:
      1. All letters (capital and lowercase), and
      2. Numbers.

NOTE: If the beneficiary does not have an SSN, then Both the, “There is no Social Security number or alternate ID”, and the, “I verify that there is no Social Security number or alternate ID associated with this claimant”, checkboxes Must be checked to continue the claim creation process in BM.

Figure: Claimant ID checkboxes for beneficiaries without SSNs.

Figure: Beneficiary biographical information section

II.3.A.2.d

Creating a profile – Contact information

The user enters the following biographical information in the Email and phone numbers section of the Create a profile window to create a new beneficiary profile.

  1. Email address:  This field is required if an email address is provided on the application. If entered,
    1. Must be between three (3) and 60 characters.
    2. Must contain the @ symbol and a period.
    3. If the email address provided is longer than 60 characters AND Email is selected as the preferred contact method, the user must develop to the beneficiary for an alternate email address or a different preferred contact method. 
    4. If the email address provided is longer than 60 characters AND Email is not selected as the preferred contact method, the user may leave this field blank.
  2. Mobile phone number:  This field is required if a mobile phone number is provided on the application.  If entered, can be a maximum of 17 numeric characters.
  3. Home phone number:  This field is required if a home phone number is provided on the application.  If entered, can be a maximum of 17 numeric characters.
  4. International number:  If an international phone number is provided, the user selects the International number checkbox that corresponds to the type of number (mobile or home). The entry must include the appropriate country code. 
  5. Preferred contact method:  This field is required. Select one of the following dropdowns:
    1. Mobile phone,
    2. Home phone, or
    3. Email.

The user enters the following biographical information in the Mailing address section of the Create a profile window to create a new beneficiary profile.

  1. Address type:  This field is required.
    1. User selects one of the following buttons:
      1. Domestic,
      2. Foreign, or
      3. Military Overseas.
    2. If Domestic is selected, the following fields display:
      1. Address line 1:  This field is required.  A maximum of 40 characters are permitted.
      2. Address line 2:  This field is not required.  A maximum of 40 characters are permitted.
      3. Address line 3:  This field is not required.  A maximum of 40 characters are permitted.
      4. City/Town/Locality:  This field is required.  A maximum of 25 characters are permitted.
      5. State:  This field is required.  User selects the appropriate state from the dropdown menu.
      6. Postal Code:  This field is required.  A maximum of 25 characters are permitted.
    3. If Foreign is selected, the following fields occurs:
      1. Address line 1:  This field is required.  A maximum of 40 characters are permitted.
      2. Address line 2:  This field is not required.  A maximum of 40 characters are permitted.
      3. Address line 3:  This field is not required.  A maximum of 40 characters are permitted.
      4. City/Town/Locality:  This field is required.  A maximum of 25 characters are permitted.
      5. State/County/Province:  This field is required.  A maximum of 25 characters are permitted.
      6. Postal Code:  This field is required.  A maximum of 25 characters are permitted.
      7. Country:  This field is required.  This field is required.  User selects the appropriate country from the dropdown menu.

Note: If ZZ (Unknown) is selected, user must enter between two (2) and 40 characters in the Other field.

    1. If Military overseas is selected, the following fields occurs:
      1. Address line 1:  This field is required.  A maximum of 40 characters are permitted.
      2. Address line 2:  This field is not required.  A maximum of 40 characters are permitted.
      3. Address line 3:  This field is not required.  A maximum of 40 characters are permitted.
      4. APO/FPO/DPO:  This field is required.  User selects the appropriate value from the dropdown menu:
        1. Army Post Office (APO),
        2. Fleet Post Office (FPO), or
        3. Diplomatic Post Office (DPO).
      5. AE/AA/AP:  This field is required.  User selects the appropriate value from the dropdown menu:
        1. Armed Forces Europe (AE),
        2. Armed Forces Americans (AA), or
        3. Armed Forces Pacific (AP).
      6. Postal Code:  This field is required.  A maximum of 25 characters are permitted.

Figure: Contact information section

II.3.A.2.e

Address validation

When the user enters a mailing address for the beneficiary, the address information is validated against VA Profile, the source for all beneficiary contact information.

If the address passes validation, beneficiary profile creation continues.

If the address fails validation, one of the following options will display:

  1. Suggestion: If the validation results in a suggested mailing address, the suggestion displays with the option to either:
    1. Use this suggested address, or
    2. Keep the address as entered.
  1. No Suggestion: If the validation results in no suggested mailing address, the only option is to Keep the address as entered.

Topic 3: Known Issues

Introduction

Several issues have been identified in BM that require users to take alternate steps to successfully process a claim. This topic contains those issues and necessary steps.  These issues will be updated as corrections are made to the system.

Change Date June 24,  2025

II.3.A.3.a

Address validation

VCEs must validate the mailing address for every claim worked in BM. This must be done before any claim authorization can be done. Failing to do so will result in the letter failing to send correctly.

Upon working any claim in BM, the user must select the Edit button in the Contact information section of the Beneficiary information screen to open the Edit contact information window to ensure that the system runs the required address validation. Users follow the steps found in M22-4, Part II, 3.C.1.d and must be sure to select the Save button upon completion.

II.3.A.3.b

TATU with kicker

BM cannot accurately process an enrollment for a top-up enrollment with a kicker. If the user finds this scenario, the claim must be processed in BDN. See Procedural Advisory: Benefits Manager Rollout Processing Controls Part 2 for details.

II.3.A.3.c

Chapter 30 category 1C

 BM cannot accurately determine eligibility for a Veteran who potentially qualifies for MGIB-AD under chapter 30 category 1C. If the user finds this scenario, the claim must be processed in BDN. See Procedural Advisory: Benefits Manager Rollout Processing Controls Part 2 for details.

II.3.A.3.d

Chapter 1606 eligibility effective dates

 BM cannot accurately determine eligibility for a Veteran who potentially qualifies for MGIB-SR under a D or C code with a BA code that is of the same or later effective date. If the user finds this scenario, the claim must be processed in BDN. See Procedural Advisory: Benefits Manager Rollout Processing Controls Part 2 for details.

II.3.A.3.e

High school enrollments and 5 month free entitlement

Chapter 35 claims involving high school enrollments and five-months free entitlement are always full time.  BM can only accurately accept such enrollments with full time modifiers of either 18 or 22 clock hours.  Users will need to take the following actions for these enrollments to ensure BM accurately calculates entitlement use.

  1. If the full-time modifier is either 18 or 22 clock hours and the enrollment certification is for at least the number of full-time clock hours, enter the actual number of enrolled clock hours from the certification.
  2. If the full-time modifier is 18 clock hours and the enrollment certification is for 17 clock hours or less, enter the number of enrolled clock hours as 18 in BM and edit the payment letter to reflect the actual enrollment.
  3. If the full-time modifier is 22 clock hours and the enrollment certification is for 21 clock hours or less, enter the number of enrolled clock hours as 22 in BM and edit the payment letter to reflect the actual enrollment.
  4. If the full-time modifier is other than 18 or 22 clock hours, update the full-time modifier to 18 clock hours and enter the number of enrolled clock hours as 18 in BM.  Edit the payment letter to reflect the actual enrollment.

II.3.A.3.f

No address in VADIR

BM cannot currently create a beneficiary if an address does not exist in the corresponding VADIR record. When this occurs, users will receive an error reading “An API error occurred” followed by an incident ID code upon attempting to create the beneficiary in BM. In this case, users will take the following steps until a system fix can be implemented.

  1. Log into DGI and search VADIR for the beneficiary.
  2. Create the claimant in DGI by completing the BIO screen (as is done to establish TOE dependents before setting up the transfer of entitlement for a Veteran).
  3. Return to BM to process the claim.

II.3.A.3.g

No training time on graduate enrollment certifications

The DGIB Managed Service (MS) is transmitting non-chapter 33 enrollment certification documents (1999s)  to BM without including certified training time, when no BDN master record exists for the beneficiary.

It is suggested that all non-chapter 33 first awards, involving graduate programs, be processed in BM, when not limited by rollout conditions detailed in Procedural Advisory: Benefits Manager Rollout Processing Controls Part 2. If the enrollment is processed through BM, and the term that is automatically pulled from Enrollment Manager (EM) is accepted, the claim can be processed without any additional action.  The term pulled in from EM can be taken as accurate, since it comes directly from the SCO.

If any adjustments are made to the enrollment, or if the claim is processed in BDN, follow local procedures to have the 1999 from EM recaptured into the TIMS file.  The recapture will pull in the training time from EM. 

Note: No recapture is needed if the number of enrolled hours is considered full time, based on the undergraduate training rate for the school.

II.3.A.3.h

Invalid chapter 30 kicker code

If the DoD reports an invalid chapter 30 kicker code for a Veteran (i.e. code did not exist as of date of enlistment, or was worth $0), Benefits Manager will be unable to make an eligibility decision. Additionally, the chapter 30 kicker table will not appear on the Entitlement and kickers page.

To resolve this issue, click the "Add VA eligibility" button on the Eligibility page, add a new suspended period, reason "Delimiting date gap", with a begin date of the current date and no end date, save the suspended period, navigate to the Entitlement and kickers screen, and edit or delete the chapter 30 kicker, as appropriate. After edits have been made to the kicker data, navigate to the Eligibility page and delete the suspended VA eligibility status previously created.
