Topic 1: Beneficiary information

Introduction

This topic discusses the Beneficiary information screen that displays upon locating a beneficiary’s profile in BM.

Change Date

June 27, 2025

II.3.C.1.a

Claimant banner

The claimant banner on the beneficiary’s profile page and contains the information displayed below.

  1. Name: The beneficiary’s name displays in first name, middle name, last name format.
  2. ID:  The beneficiary’s primary identifier displays either in social security number (SSN) format (XXX-XX-XXXX) or in claim number (C #) format (CXX-XX-XXXX).
  3. Last benefit used:  This field displays the most recent non-chapter 33 benefit authorized in BM.  Possible values are:
    1. Chapter 1606 (MGIB-SR),
    2. Chapter 30 (MGIB-AD),
    3. Chapter 35 (DEA), or
    4. N/A

Note: “N/A” only displays if the beneficiary has no previously authorized claims in BM.

  1. Six-credit hour exclusion:  This field displays the six-credit hour exclusion as used or is unused, as appropriate.
  2. Station:  This field displays the Regional Processing Office (RPO) of jurisdiction for the beneficiary.  It will display the most recently saved station. This field can be adjusted, if necessary, by changing the station in the dropdown menu.

Figure: Claimant banner

II.3.C.1.b

Beneficiary information screen – General

The default view of the beneficiary’s profile page shows the Beneficiary information screen, which is divided into the following sections:

  1. Biographical information,
  2. Contact information,
  3. Other,
  4. Payment address,
  5. Payment status, and
  6. Relationship information.

Note: The Relationship information section only displays on profiles of chapter 35 beneficiaries with verified relationship(s).

To make changes to the beneficiary’s information, the user selects the Edit button in the appropriate section.

Figure: Beneficiary information screen

II.3.C.1.c

Biographical information

The Biographical information section of the Beneficiary information screen displays name, gender, date of birth (DOB), and date of death.  This information is stored by VA in the Master Person Index (MPI).  When MPI is updated, the new information will display in BM. 

To make changes to biographical information in BM, the user selects the Edit button to open the Edit biographical information window which allows the editing of the following information.

  1. First name:  This is a required field.
    1. Must be between one (1) and 40 characters.
    2. Allows the following characters:
      1. All letters (capital and lowercase),
      2. Space,
      3. Hyphen, and
      4. Apostrophe.
    3. When only one (1) character is entered, it only accepts uppercase A through Z.
  2. Middle name:  This is not a required field but must be entered if provided.
    1. If entered, must be between one (1) and 40 characters.
    2. Allows the following characters:
      1. All letters (capital and lowercase),
      2. Space,
      3. Hyphen, and
      4. Apostrophe.
    3. When only one (1) character is entered, it only accepts uppercase A through Z.
  3. Last name:  This is a required field.
    1. Must be between one (1) and 40 characters.
    2. Allows the following characters:
      1. All letters (capital and lowercase),
      2. Space,
      3. Hyphen, and
      4. Apostrophe.
    3. When only one (1) character is entered, it only accepts uppercase A through Z.
  4. Suffix:  This is not a required field but must be entered if provided.
    1. If entered, must be between one (1) and 40 characters.
    2. Allows the following characters:
      1. All letters (capital and lowercase),
      2. Numbers, and
      3. Period.
  5. Date of birth:  This is a required field.
    1. Must be in MM/DD/YYYY format.
    2. User can either type in the date or select the appropriate date from the calendar icon.
  6. Date of death:  This is a required field, if information regarding a date of death is on record.  This field can be populated either by automated MPI data ingestion or manual entry by the user.
    1. Must be in MM/DD/YYYY format.
  7. Social Security number:  This field is required if an SSN is available for the beneficiary.
    1. If no SSN was provided when the beneficiary profile was created, the user can edit this field.
    2. If an SSN is already on file for the beneficiary, this field cannot be edited from the Edit beneficiary information window.
    3. This field must be nine (9) digits and accepts only numeric characters.
  8. File number / Alternate ID:  This field is required if no SSN is on file.
    1. If no identifier is entered, the user can edit this field.
    2. If data is received from the MPI, this field cannot be edited from the Edit beneficiary information window.
    3. This field must be either eight (8) or nine (9) digits and accepts only numeric characters.
  9. Gender: This field is required, if provided.  The user selects the appropriate value from the dropdown menu with the following options:
    1. Male,
    2. Female, or
    3. Unspecified.

The user selects the Save button to edit the biographical information.  Users also can use the Cancel or X button to close the Edit biographical information window without taking any actions.

Figure: Edit biographical information window  

If an Education Service (EDU) user receives information that updates the Biographical information for an existing beneficiary and MPI does not have the updated information (as visible in Share), follow local procedures to submit a Service Now (SNOW) ticket with the appropriate documentation.  See M22-4, Part II, 1.A  for details on procedures and required documents to have updates made to MPI.

II.3.C.1.d

Contact information

To validate a beneficiary’s contact information, the user selects the Edit button to open the Edit contact information window which allows the following actions.

  1. If the mailing address is valid, the user selects the Save button.
  2. If the mailing validation fails, the user can either adjust the address to resolve the error or keep the address as entered.
  3. If the mailing validations fails and a similar address is found, the user can either use the selected address or keep the address as entered.
  4. The user must select the Save button to exit the Edit contact information window and return to the Beneficiary information screen.

VCEs must validate the mailing address for every claim worked in BM. This must be done before any claim authorization can be done. Failing to do so will result in the letter failing to send correctly. See M22-4, Part II, 3.A.3.a for details.

  1. Email address:  This field is required if an email address is provided on the application. If entered,
    1. Must be between 3 and 60 characters.
    2. Must contain the @ symbol and a period.
    3. If the email address provided is longer than 60 characters AND Email is selected as the preferred contact method, the user must develop to the beneficiary for an alternate email address or a different preferred contact method. 
    4. If the email address provided is longer than 60 characters AND Email is not selected as the preferred contact method, the user may leave this field blank.
  2. Mobile phone number:  This field is required if a mobile phone number is provided on the application.  If entered, can be a maximum 17 numeric characters.
  3. Home phone number:  This field is required if a home phone number is provided on the application.  If entered, can be a maximum 17 numeric characters.
  4. International number:  If an international phone number is provided, the user selects the International number checkbox that corresponds to the type of number (mobile or home). The entry must include the appropriate country code.
  5. Address type:  This field is required.  See M22-4, Part II, 3.A  for details on address validations in BM.
    1. The user selects one of the following buttons:
      1. Domestic,
      2. Foreign, or
      3. Military Overseas.
    2. If Domestic is selected, the following fields display:
      1. Address line 1:  This field is required.  A maximum of 40 characters are permitted.
      2. Address line 2:  This field is not required.  A maximum of 40 characters are permitted.
      3. Address line 3:  This field is not required.  A maximum of 40 characters are permitted.
      4. City/Town/Locality:  This field is required.  A maximum of 25 characters are permitted.
      5. State:  This field is required.  The user selects the appropriate state from the dropdown menu.
      6. Postal Code:  This field is required.  A maximum of 25 characters are permitted.
    3. If Foreign is selected, the following fields display:
      1. Address line 1:  This field is required.  A maximum of 40 characters are permitted.
      2. Address line 2:  This field is not required.  A maximum of 40 characters are permitted.
      3. Address line 3:  This field is not required.  A maximum of 40 characters are permitted.
      4. City/Town/Locality:  This field is required.  A maximum of 25 characters are permitted.
      5. State/County/Province: A maximum of 25 characters are permitted.
      6. Postal Code:  This field is required.  A maximum of 25 characters are permitted.
      7. Country:  This field is required.  This field is required.  The user selects the appropriate country from the dropdown menu.

Note:  If ZZ (Unknown) is selected, user must enter between 2 and 40 characters in the Other field.

    1. If Military overseas is selected, the following fields display:
      1. Address line 1:  This field is required.  A maximum of 40 characters are permitted.
      2. Address line 2:  This field is not required.  A maximum of 40 characters are permitted.
      3. Address line 3:  This field is not required.  A maximum of 40 characters are permitted.
      4. APO/FPO/DPO:  This field is required.  The user selects the appropriate value from the dropdown menu:
        1. Army Post Office (APO),
        2. Fleet Post Office (FPO), or
        3. Diplomatic Post Office (DPO).
      5. AE/AA/AP:  This field is required.  The user selects the appropriate value from the dropdown menu:
        1. Armed Forces Europe (AE),
        2. Armed Forces Americans (AA), or
        3. Armed Forces Pacific (AP).
      6. Postal Code:  This field is required.  A maximum of 25 characters are permitted.
  2. Preferred contact method:  This field is required.  Once an option is selected, an entry is required within the corresponding field.  The user selects one of the following dropdown options:
    1. Mobile phone,
    2. Home phone, or
    3. Email.
  3. Enrollment verification method:  This field is required if a preference is indicated.  The user selects one of the following dropdown options:
    1. Text,
    2. Email, or
    3. None.
  4. Opt-in status:  This field is required if an enrollment verification opt-in preference is indicated.  The user selects one of the following dropdown options:
    1. Yes, or
    2. No.
  5. Parent/guardian contact information:  User checks this checkbox if the beneficiary is under 18 years old or has a permanent guardian.  If selected, the user enters the following parent/guardian information following the rules previously listed:
    1. First name,
    2. Last name,
    3. Address type,
      1. Domestic
      2. Foreign
      3. Military overseas
    4. Mobile phone number,
    5. Home phone number,
    6. Email address, and
    7. Preferred contact method.

User selects the Save button to edit the contact information.  Users also can use the Cancel or X button to close the Edit contact information window without taking any actions.

Figure: Edit contact information window

II.3.C.1.e

Other information

To edit the following fields in the Other section, user selects the Edit button to open the Edit other window.

  1. Education level: This field is automatically populated from the beneficiary’s VA/DoD Identity Repository (VADIR) record, if applicable.  The user can edit this field by selecting the appropriate value from the dropdown menu.
  2. High school graduation date:  This field is required if a high school graduation date is provided.  For beneficiaries under 18 years old, this field can impact calculated payments.
    1. Must be in MM/DD/YYYY format.
    2. User can either type in the date or select the appropriate date from the calendar icon.
    3. If the beneficiary only provides the month and year of graduation, the user enters a graduation date of the last day of the given month.  If there is a potential impact to payments, the user must develop for the exact graduation date.
  3. Power of Attorney: This field is required if the beneficiary has a power of attorney (POA) on file in VADIR or has submitted a POA form.
    1. User enters the 3-character alphanumeric POA code corresponding to the beneficiary’s POA.  This will display a read-only Power of Attorney name field.
    2. If codes “099” or “066” are entered, the Power of Attorney name field can be edited with the appropriate POA information up to a maximum of 200 characters.
  4. Six-credit hour exclusion:  This field is required if the beneficiary has used the six-credit hour exclusion in an EDU system other than BM, such as the Chapter 33 Processing System (DGI) and BDN.
    1. Must be in MM/DD/YYYY format.
    2. User can either type in the date or select the appropriate date from the calendar icon.
  5. The user selects the Save button to save the other information.  Users also can use the Cancel or X button to close the Edit other window without taking any actions.

Figure: Edit other window

II.3.C.1.f

Adding a payment address

For claims processed in BM, the official system of record for payment addresses is the Corporate Database (CorpDB) which is shared across VBA.  Education processors do not have direct access to CorpDB but can view information from the CorpDB in SHARE and update and view payment addresses in DGI. 

Beneficiaries may only use one payment address for all VA benefits.  Any changes made by EDU to a beneficiary’s payment address will change their payment address for all VA business lines.  Be sure the most recent payment address is always used.

To add the beneficiary’s payment address, the user selects the Add button to open the Add payment address window, allowing the user to enter the following information.

  1. Payment address type:  This field is required if a payment address is present.  The user selects the appropriate option from the dropdown menu.  Options are:
    1. Direct deposit, or
    2. Physical address.
  2. If direct deposit is selected, the following information can be entered.
    1. Account type:  This field is required when a direct deposit payment address is used. The user selects the appropriate option from the dropdown menu.  Options are:
      1. Checking, or
      2. Savings.
    2. Routing number:  This field is required when a direct deposit payment address is used.  Data entry rules are as follows:
      1. Field must be 9 digits.  Only numeric characters are accepted.
      2. An invalid routing number will display an error message.
    3. Account number:  This field is required when a direct deposit payment address is used.  Field must be between five (5) and 17 digits.  Only numeric characters are accepted.

Note: Only domestic United States EFT addresses can be used for EDU direct deposit payments.  No foreign EFT payment addresses can be entered into BM.

Note: If a beneficiary is using an International Direct Deposit (IDD) payment address for a non-EDU benefit, they can continue to do so.  In this case, all EDU benefits will be paid via a paper check sent to their mailing address until such a time that EDU can handle IDD payment addresses.  If EDU updates the payment address in BM, all payment addresses, including the IDD, will be updated to the new address.  If the beneficiary wants to continue using their IDD to receive non-EDU benefits, EDU users should not make any changes to the payment address in BM.

  1. If physical address is selected, user must answer the question: Does the mailing address match the payment address?
    1. If Yes is selected, then the existing mailing address on the BM Contact information section will display as the payment address.
    2. If No is selected, the user enters the new physical address to where payments will be sent.  The rules for address entry are the same as the rules in M22-4, Part II, 3.B.1.d.

The user selects the Add button to save the payment information.  Users also can use the Cancel or X button to close the Add payment address window without taking any actions.

Figure: Add payment address window

Users should enter the payment address provided on any original application when reviewing the Beneficiary information screen since eligibility cannot be determined with certainty until the claim is fully processed.  BM is designed to automatically remove payment addresses for beneficiaries whose claim is denied.  As such, upon authorization of a non-chapter 33 denial, the payment address will be removed automatically and will display as blank.

II.3.C.1.g

Editing a payment address

If a beneficiary has a payment address in CorpDB (from an earlier EDU claim or from another business line), then the most recent address displays in the Payment address section of the Beneficiary information screen.  A Last updated field also displays the following information:

  1. The user/source of the most recent update,
  2. The date of the most recent update, and
  3. The time of the most recent update.

To edit a payment address for a beneficiary who already has one established, the user selects the Edit button to open the Edit payment address window, allowing the user to change any of the information listed in M22-4, Part II, 1.B.1.f.

The user selects the Save button to save the payment information.  Users also can use the Cancel or X button to close the Edit payment address window without taking any actions.

If the beneficiary has a fiduciary on file with CorpDB, the user cannot edit the payment address.  If a change to an existing fiduciary is required, the user needs to contact the Fiduciary Hub to request an update.  See (I) 2022 14 September: Procedural Advisory: Changing Contact or Direct Deposit Information When a Beneficiary has a Fiduciary Assigned for full details on contacting the Fiduciary Hub.

If the beneficiary has a Change of Address Block (also referred to as Fraud Flash or 02X) on file with CorpDB, the user cannot edit the payment address.  If a change to an existing address is required, the user needs to contact the SLC VBA Incident team at VAVBAWAS/CO/FRAUDINCIDENTTEAM to request an update. A temporary removal of the block may also be necessary during initial authorization of a claim in Benefits Manager. 

If a beneficiary has a payment address in CorpDB (from an earlier EDU claim or from another business line), then the most recent address displays in the Payment address section of the Beneficiary information screen.

All past payment addresses can be viewed using the Previous payment address button to open the Previous payment address window.

II.3.C.1.h

Payment status

Since BDN was a processing system as well as a payment system, it was able to stop payments in some circumstances to prevent certain erroneous payments.  eMPWR does not have this ability since it is only a payment system.  The Payment status section of the Beneficiary Information screen allows a beneficiary’s account to be placed into suspense, preventing the release of certain payments until the suspense is resolved.  Reasons for a file to be suspended include the return of payments as undeliverable, report of a date of death, or a potential administrative error.

The “Claimants in Suspense” dashboard provides the list of suspended claims by benefit type, age, Regional Processing Office (RPO), and reason so that RPO and EDU leadership can keep track of suspended claims.

Managing payment status

To update the beneficiary’s payment status, the user selects the Edit button of the Payment status section of the Beneficiary Information screen to open the Payment status window.  This allows the user to manage the following information.

  1. Status:  This field is required.  The user selects from the following options in the dropdown menu:
    1. Active (this is the default value that permits payments to be sent to eMPWR), or
    2. Suspended.
  2. Reason for suspension:  This field is required if Status of suspended was selected.  User selects from the following options in the dropdown menu:
    1. Beneficiary payment undeliverable,
    2. Date of death in system,
    3. Potential administrative error, or
    4. Other (if selected, the user must enter a Description).

User selects the Save button to save the payment status information.  Users can also use the Cancel or X button to close the Payment status window without taking any actions.

When the situation requires a suspended file to be returned to active status, a warning message displays reading “Changing the payment status to active will release all suspended payment transactions.  Confirm you wish to proceed.”  To move forward the user must select the Proceed with this action checkbox and select Save.

Note: Returning the Payment status to Active from Suspended will release all matured payment and debt transactions held up by the suspense.

Note: An information message displays on all screens for a beneficiary with a suspended file indicating the file is in suspense and the corresponding reason.

Working a suspended file

Although users should resolve a suspense issue before finishing a claim in most cases, it is possible a file will stay in suspense for a longer period and processing will be required while the file is in suspense. BM allows users to create claims for files that are in suspense.  For details on authorizing a claim, see M22-4, Part II, 3.F.

II.3.C.1.i

Relationship information

The Relationship information section of the Beneficiary information screen displays only for chapter 35 beneficiaries with verified relationships and provides read-only information.  Relationships can only be edited within a claim.

The Relationship information section displays the following information.

  1. Relationship type:  This field has the possible values of:
    1. Child
      1. Child,
      2. Stepchild,
      3. Adopted child.
    2. Spouse.
  2. Relationship to: This field lists the veteran’s name in first name, middle name, last name format.

The Relationship information section can be expanded to see the following details in addition to the information mentioned above.

  1. Veteran’s date of birth:  This field displays in MM/DD/YYYY format.
  2. Veteran’s Social Security number / file number
  3. Veteran’s date of death:  This field only displays if applicable to the Veteran and shows in MM/DD/YYYY format.
    1. Service connected:  This field has possible values of Yes or No.
    2. In line of duty:  This field has possible values of Yes or No.
  4. Marriage date:  This field only displays if the Relationship type is Spouse and shows in MM/DD/YYYY format.
  5. Remarriage date:  This field only displays if the Relationship type is Spouse and applicable to the claim.  It shows in MM/DD/YYYY format.
  6. Divorce date:  This field only displays if the Relationship type is Spouse and applicable to the claim.  It shows in MM/DD/YYYY format.
  7. Divorce fault indicator:  This field only displays if the Relationship type is Spouse and applicable to the claim. Possible values are Yes or No.
  8. Adoption date: This field only displays if the Relationship type is Adopted child and shows in MM/DD/YYYY format.
  9. Parents’ marriage date:  This field only displays if the Relationship type is Stepchild and shows in MM/DD/YYYY format.
  10. Parents’ divorce date:  This field only displays if the Relationship type is Stepchild and applicable to the claim.  It shows in MM/DD/YYYY format.
  11. Veteran is providing at least 50% care for dependent: This field only displays if the Relationship type is Stepchild and applicable to the claim.  Possible values are Yes or No.

Topic 2: Enrollment verification

Introduction

This section discusses enrollment verification in BM.

Change Date

October 16, 2024

II.3.C.2.a

Enrollment verification methods

Beneficiaries have several ways to verify their enrollment.

  1. Text messaging: The preferred method monthly verification is text messaging. Beneficiaries with a U.S. mobile phone number (including Puerto Rico) can use text messaging to verify enrollment. Beneficiaries who provide a valid mobile phone number will receive a text message inviting them to “opt-in” to text message verification. Those who reply Yes can then verify their monthly enrollment via text message. Those who reply No will be defaulted to email, if a valid email address is in BM. The link within that invitation text message will expire in 14 calendar days. 

a.  Opted-in to text: Beneficiaries will receive the first text verification message on the last day of the month.

  1. If the beneficiary selects Yes, their enrollment will be verified, payment released automatically by BM, and an ATTENDY document will be filed in the Image Management System (TIMS). Once verified a confirmation text is sent to the beneficiary.
  2. If the beneficiary selects No, no payments will be released and an ATTENDN document will be filed in TIMS, creating an active token for user review.
  3. If no response is received after three (3) days, a follow-up text message is sent and the link to verify the enrollment will remain open for three (3) additional calendar days.  After seven (7) calendar days, the text conversation is closed, and the system is updated with No response.  No payments will be released and an ATTENDR document will be filed in TIMS, creating an active token for user review.

b. Opted-out of Text:  Beneficiaries will be automatically enrolled in email verification and will receive an email message as outlined in the subsequent section.

  1. Email: Beneficiaries can also verify their enrollment information each month through an automated email system. If a beneficiary opts-out of text messages, they will be automatically enrolled into the email system, if a valid email address is in the BM.
    1. The TIMS document types for email verification are the same as the text message document types.
    2. Beneficiaries will be sent an automated email on the last day of the month that needs to be responded to with either "Yes, my enrollment is the same" or "No, my enrollment has changed"
    3. The beneficiary will be sent a reminder email after seven (7) calendar days with no response.  The conversion will close after 14 calendar days without a response.
    4. If the beneficiary selects Yes, their enrollment will be verified, payment released automatically by BM, and an ATTENDY document will be filed in TIMS. The beneficiary will be sent to a VA.gov webpage displaying a successful verification message.
    5. If the beneficiary selects No, no payments will be released and an ATTENDN document will be filed in TIMS, creating an active token for user review. The beneficiary will be sent to a VA.gov webpage asking them to contact their School Certifying Official (SCO) to adjust their enrollment if necessary.
    6. After five months without enrollment verification, the beneficiary’s record will be considered delinquent and an ATTENDNR document will be filed in TIMS, creating an active token. All payments will be held until the beneficiary manually verifies their enrollment by calling the Education Call Center (ECC).
    7. If the beneficiary clicks an expired link, they will be sent to a VA.gov webpage informing them that no action was taken because the link has expired and directs them to contact the ECC.
  2. Telephone: The beneficiary may also call the ECC to verify their enrollment status.  ECC staff can verify the enrollment within BM by performing the following steps:
    1. Verify the beneficiary’s identity using standard protocol.
    2. If the beneficiary verifies that their enrollment is the same (i.e., “yes”), verification may occur.
    3. If the beneficiary states that their enrollment has changed (i.e., “no”), additional action is needed.
      1. Create a Report of Contact (119E) describing any changes to the enrollment. The 119E must be captured to TIMS.
      2. The ECC agent should advise the beneficiary to notify their SCO of the enrollment status change if a 1999B – Enrollment amendment document has not been received.
      3. Route claim, following local guidance, to the appropriate supplemental queue for processing.
  3. Verify Your Enrollment (VYE) System: The beneficiary may also submit their enrollment verification through the VYE system, the replacement for the WAVE system.

Note: VYE is only available for chapter 30 and chapter 1606 beneficiaries until further notice.  Chapter 35 beneficiaries must verify their attendance through one of the other options.

  1. Other:  An email from the claimant received via Ask VA (AVA) or through any VA e-mail system may be used to verify the enrollment status.

II.3.C.2.b

Impact on payments

Enrollment verification affects EDU payments (i.e., basic and/or kicker payments) made to beneficiaries.  As a result of the enrollment verification requirements, BM creates a separate “35” lump sum business transaction (BT) for each month, where a basic and/or kicker payment is due.

  1. Unlike the Post-9/11 GI Bill (chapter 33), these monthly basic and/or kicker payments are NOT released automatically by BM for the first two months.
  2. The beneficiary must supply a positive (i.e., “Yes, my enrollment is the same”) monthly verification response for payment to be released.
  3. If the beneficiary responds “No, my enrollment has changed” or fails to verify for five consecutive months, basic and/or kicker payments will not be released and VCE will need to take appropriate action.

II.3.C.2.c

Processing monthly verifications

Claims processing impact and responsibilities.

Users should continue to process enrollment certifications (VA Form 22-1999) and notices of change in student status (VA Form 22-1999b) submitted using current guidance (see M22-4, Part II, 3.E for details).  If an “unchanged” amendment is received, it should be accepted from Enrollment Manager (EM) or manually entered into BM. The BM claim should be authorized with payments and letter(s) should be released.

BM has been updated to automatically accept text messages which state Yes. These affirmative text messages will NOT be sent to TIMS.  BM will automatically update the enrollment verification screen and release payment.

If the beneficiary responds No to the text message, BM will automatically hold all basic and kicker payments. An ATTENDN TIMS document will be created as an active claim token and routed to the appropriate supplemental queue for routine processing. When the user receives the ATTENDN document, the following actions will be performed.

  1. Review the TIMS folder for any reported changes from the school.
  2. Process any pending enrollment certifications or adjustments found, as usual.
    1. If there is no evidence of a change submitted by the school in the TIMS folder, the user must contact the school to request any updated enrollment information and update TIMS with a 119E describing any changes or confirmation that the beneficiary remains certified correctly.
  3. Process reported changes documented in TIMS.
  4. Ensure the BM system-generated letter is not suppressed.
  5. After authorization, or if no changes reported, the enrollment verification screen must be updated.
    1. Remember the “ATTENDN” represents the beneficiary stating that the enrollment is not verified as certified.
    2. If a first award is authorized and any Basic or Kicker awards are due, the authorizer must update the Enrollment verification screen accordingly.

Note: If the school confirms over the phone that a change in enrollment was made, the user can process the change based on the telephone contact but must ensure the 119E documenting the action is captured in the TIMS folder. (The user should remind the school that a proper submission of enrollment or change in enrollment is required for compliance purposes.)

How to verify enrollments in BM

Once an enrollment has been verified by the beneficiary, the Enrollment verification screen must be updated. This screen is accessed through the Beneficiary information screen and is handled outside a claim.

  1. Navigate to the Enrollment verification screen to update to the new Verified through date.
  2. Select the period(s) which have been verified from those listed under Unverified periods.
  3. Select the Submit verification button. 

Figure: Enrollment verification tab

II.3.C.2.d

Opt-in methods

Opt-in Method (Manual Updates): Education Call Center Technicians (ECCT) will assist with updates to a beneficiary’s preference of their enrollment verification messaging method when assistance is requested by the caller by performing the following steps.

  1. Verify the beneficiary’s identity using standard protocol. 
  2. Ensure the mobile phone number field under Contact information on the Beneficiary information screen reflects a mobile number (required to properly receive the text messages).
  3. Verify and update, when necessary, the beneficiary’s email address under Contact information (required for email communications).
  4. Review the current Enrollment verification method on the Beneficiary information screen, which may have been updated automatically. If the current setting reflects the preference of the beneficiary, no additional action is required.
  5. When the current Enrollment verification method requires updating, the user must document the opt-in or opt-out request in TIMS with a Report of Contact form (119E).
    1. If there is no open claim token when adding the 119E to TIMS, use the COPYA document type.
    2. After capture of the 119E is complete, change the document type to 119 to prevent opening a claim token in error.
  6. There are three manual update settings which address any scenario. Apply one of these three Opt-In Method settings below:
    1. Opt-in to text: If the beneficiary calls the ECC to opt in to receive text messages, the ECCT will navigate to the Beneficiary information screen and manually change the Enrollment verification method field under Contact information to Text and the Opt-in status field to Yes. This will allow the beneficiary to receive enrollment verification text messages each month. 
    1. Opt-out of text (via enroll in email): If the beneficiary calls the ECC to opt out from text messages, the ECCT will navigate to the Beneficiary information screen and change the Enrollment verification method to Email and the Opt-in status field to Yes. The ECCT must confirm there is a valid email address on the Beneficiary information screen in BM. This will allow the beneficiary to receive enrollment verification emails each month. This is the default automatic setting for opted-out of text.

Note: If there is not a valid email provided, follow the Opt-out text of and email process below when the beneficiary has also opted-out of text

    1. Opt-out of text and email:  If the beneficiary calls the ECC to opt out of receiving text messages and emails or does not have a valid email, the ECCT will navigate to the Beneficiary information screen and change the Enrollment verification method to Email and the Opt-in status field to No. This will ensure the beneficiary will no longer receive enrollment verification text messages or emails each month.

Note: The ECCT will advise the beneficiary they must call the ECC to verify their monthly enrollment when opted out of both text and email.

  6. Monthly payment: This field displays the total amount of the award being paid for that period of training, including both basic and kicker rates. In United States Dollars, including two decimal places.

Delinquent verification

If a beneficiary fails to respond to verification requests for five (5) consecutive months, they are placed in a delinquent status. An ATTENDNR document will be created and filed TIMS as an active claim token.

  2. The user creates and sends a letter to the beneficiary informing them of the delinquency. The user must capture the letter into TIMS. 
  3. Finish the TIMS claim token.

Note: A monthly delinquent certification report will be sent to each of the RPOs for action.

II.3.C.2.j

Invalid contact information

Expanded functionality of email verification requires no action when the Enrollment verification method has been automatically set to Email with an Opt-in status of Yes, indicating the beneficiary will be receiving email enrollment verification messages.

Note: Due to system limitations, phone numbers that originate either from a foreign country or U.S. territory (excluding Puerto Rico) are displayed as invalid and will not allow text message use for enrollment verification purposes.

  2. Review the TIMS folder for a new mobile phone number or email.

The user must capture the letter into TIMS.  The user also must send an email the appropriate SCO informing them of this issue.  The email must be captured into TIMS.
    3. Finish the TIMS claim token.

Topic 3: Claim history and creation

Introduction

This topic discusses the Claim history screen and the steps to create a claim in BM.

Change Date

October 16, 2024

II.3.C.3.a

General

By selecting the Claim history screen of the Beneficiary information screen, the user can access a summary of the beneficiary’s claim history.  This screen displays the following information for all in-progress and authorized claims.

  1. Claim date
    1. Date entered upon claim creation in MM/DD/YYYY format.
  2. Benefit
    1. Possible values are:
      1. Chapter 1606,
      2. Chapter 30, or
      3. Chapter 35.
  3. Adjudicator
    1. Displays the name of the most recent user who made an update to an in-progress claim or the last person to take a non-authorizing action on an authorized claim.
    2. If the most recent updates were done automatically, this will display the system name.
  4. Authorizer
    1. Displays the name of the user who authorized the given claim.
    2. If the claim was authorized automatically, this will display the system name.
  5. Authorization date
    1. Displays the date the claim was authorized in MM/DD/YYYY format.
  6. Claim status
    1. Possible values are:
      1. In progress,
      2. Authorized, or
      3. Deleted.

The Claim history can be expanded to see the following details in addition to the information mentioned above.

  1. Entitlement source
    1. Possible values are:
      1. Self, or
      2. Name and social security number (SSN) of Veteran source.
  2. Relationship type (if applicable)
    1. If the relationship is not Self, this field displays the relationship between the beneficiary and the Veteran.  Possible values are:
      1. Spouse,
      2. Child,
      3. Stepchild, or
      4. Adopted Child.
  3. Description
    1. This shows the description entered at the time of claim creation.  If the description is left blank, this field will display “N/A.”

Figure: Claim history tab

II.3.C.3.b

Claim history actions

From the Claim history screen, the user can take several actions depending on the claim status.

If the claim is In progress, the user can take the following actions by selecting the corresponding icon for the given claim.

  1. The Edit claim information window is accessed by selecting the pencil icon for the claim.  This allows users to adjust the Claim date and the Description, as necessary.
  1. The details for the claim are accessed by selecting the view icon for the claim.  This takes the user to the Eligibility screen and allows for updates to be made to the claim.
  2. A claim can be deleted by selecting the Delete claim button in the expanded claim view.

If the claim is Authorized, the user can take the following actions by selecting the corresponding icon for the given claim.

  1. The Edit claim information window is accessed by selecting the pencil icon for the claim.  This allows users to adjust only the Description, as necessary.
  1. The details for the claim are accessed by selecting the view icon for the claim.  This takes the user to the Claim summary screen without the ability to make changes to the claim.

II.3.C.3.c

Claim creation

A claim in BM is the unit of work required to establish eligibility, generate payments or debts, and to complete all other in-system adjudicative actions for non-chapter 33 benefits.  There can only be one in-progress claim for a beneficiary at a time.

All claims opened in BM must have Flash message reading "BM Processing Only" added to the Image Management System (TIMS) file for the beneficiary indicating the claim can only worked in BM. 

To initiate a new claim, the user selects the Create a claim button under the claimant banner on the beneficiary profile page.  This opens the Create a claim window which allows the user to enter of the following information.

  1. Claim date: This is a required field. 
    1. Must be in MM/DD/YYYY format.
    2. The user can either manually type in the date or select the appropriate date from the calendar icon.
    3. The claim date cannot be a future date.
    4. If a claim date over one year in the past is entered, a warning message displays to alert the user, but the claim can still be created if appropriate.
  2. Benefit: This field is required.
    1. The user selects one of the following dropdown options:
      1. Chapter 1606 (MGIB-SR),
      2. Chapter 30 (MGIB-AD), or
      3. Chapter 35 (DEA).
  3. Description:  This is a free text box with a maximum of 200 characters used to describe the scenario(s) being adjudicated in the claim.

The user selects the Add claim button to create a claim.  Users also can use the Cancel or X button to close the Create a claim window without taking any actions.

If a Benefit of Chapter 1606, Chapter 30, or Chapter 35 (supplemental claims only) are selected, the user is taken directly to the Eligibility screen which allows for updates to be made to the claim.  See M22-4, Part II, 3.C for details on chapter 35, chapter 1606, and chapter 30.

If a Benefit of Chapter 35 is selected and it is an original claim, the Add claim button changes to the Continue and establish relationship button.

Selecting the Continue and establish relationship button opens the Establish relationship window which allows the user to enter the following information.

  1. Relationship type:  This field is required.
    1. User selects one of the following dropdown options:
      1. Spouse,
      2. Child,
      3. Adopted child, or
      4. Stepchild.
  2. Once a Relationship type is selected, the user selects the Continue button which displays the Veteran information field.  The user must search by Veteran ID (SSN or alternate ID) to establish the relationship.
    1. User enters either eight (8) or nine (9) digits and selects Search to search the claimant database for existing beneficiaries.
    2. If no matching result is returned from the claimant database, the user can either Restart search or to search the VA/DoD Identity Repository (VADIR) by selecting the Search VADIR button.
    3. If no matching result is returned from VADIR, the user can either Restart search of search the Master Person Index (MPI) by adding the veteran’s Last name and selecting the Search MPI button.
  3. If any of the searches returns the appropriate Veteran, the user selects the checkbox next to the Veteran’s name and selects the Add claim button and the user is taken to the Eligibility screen and allows for updates to be made to the claim.  See M22-4, Part II, 3.C for further details.
  4. If none of the searches returns the appropriate Veteran, the user can either attempt a new search or can select the Document Veteran information option to create the Veteran’s profile by entering the following information.
    1. Veteran ID:  This is a required field.
      1. Must be either eight (8) or nine (9) characters.
      2. Allows the following characters:
        1. All letters (capital and lowercase), and
        2. Numbers.
    2. First name:  This is a required field.
      1. Must be between one (1) and 40 characters.
      2. Allows the following characters:
        1. All letters (capital and lowercase),
        2. Space,
        3. Hyphen, and
        4. Apostrophe.
      3. When only one (1) character is entered, it only accepts uppercase A through Z.
    3. Middle name:  This is not a required field but must be entered if provided.
      1. If entered, must be between one (1) and 40 characters.
      2. Allows the following characters:
        1. All letters (capital and lowercase),
        2. Space,
        3. Hyphen, and
        4. Apostrophe.
      3. When only one (1) character is entered, it only accepts uppercase A through Z.
    4. Last name:  This is a required field. 
      1. Must be between one (1) and 40 characters.
      2. Allows the following characters:
        1. All letters (capital and lowercase),
        2. Space,
        3. Hyphen, and
        4. Apostrophe.
      3. When only one (1) character is entered, it only accepts uppercase A through Z.
    5. Suffix:  This is not a required field but must be entered if provided.
      1. If entered, must be between one (1) and 40 characters.
      2. Allows the following characters:
        1. All letters (capital and lowercase),
        2. Numbers, and
        3. Period.
    6. Date of birth (Veteran):  This is a required field.
      1. Must be in MM/DD/YYYY format.
      2. User can either manually type in the date or select the appropriate date from the calendar icon.
    7. Provide a date of death:  This checkbox is selected if a date of death is on file for the Veteran.  If selected, the following fields display:
      1. Date of death:  This is a required field if the Provide a date of death checkbox is selected. 
        1. Must be in MM/DD/YYYY format.
        2. User can either manually type in the date or select the appropriate date from the calendar icon.
      2. Service connected:  This checkbox is selected if the Veteran’s death was service connected.
      3. In line of duty:  This checkbox is selected if the Veteran’s death was in the line of duty.
    8. Relationship type:  This field is required and can be edited here from the earlier search value.
      1. User selects one of the following dropdown options:
        1. Spouse,
        2. Child,
        3. Adopted child, or
        4. Stepchild.
    9. Date of birth (Beneficiary):  This is a required field.
      1. Must be in MM/DD/YYYY format.
      1. User can either manually type in the date or select the appropriate date from the calendar icon.
    2. Marriage date:  This field displays only if the Relationship type is Spouse and shows in MM/DD/YYYY format.
    3. Adoption date: This field displays only if the Relationship type is Adopted child and shows in MM/DD/YYYY format
    4. Parents’ marriage date:  This field displays only if the Relationship type is Stepchild and shows in MM/DD/YYYY format.
    5. Verification checkbox:  User selects the checkbox certifying that they have verified the relationship by reviewing the relevant documents.

Once all appropriate information is added, the user selects the Add information to claim button and the user is taken to the Eligibility screen and allows for updates to be made to the claim.  See M22-4, Part II, 3.C for further details.

The user selects the Add claim button to create a claim.  Users also can use the Cancel or X button to close the Create a claim window without taking any actions.
