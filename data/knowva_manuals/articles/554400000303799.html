II.2.D.3.a

Search

The search for a beneficiary’s record starts in the chapter 33 DGI database. 

Begin the search by entering any of the following in the corresponding fields on the screen:

(1)   SSN or File #,

(a)   When searching by a social security number (SSN), enter nine (9) numeric digits without dashes.

(b)   When searching for a file number, enter the eight (8) numeric digits without spaces or dashes.

•  DGI automatically populates the “C” prefix.

For example, if you search for file number 12345678, DGI will display C12345678.

•  File number searches only apply to DGI, not to the VA/DoD Identity Repository (VADIR).

(2)   First Name or Last Name,

(a) When searching by first name, use a minimum of one (1) character and a maximum of 20. You cannot search by first name only.

(b) Searching by last name requires a minimum of two (2) characters and maximum of 26. Do not use special characters, hyphens, or spaces. If you do, they will be ignored for comparison purposes.

(c) Search is not case sensitive.

(d) If more than 100 matches are found, you will need to further refine your search.

(3)   Date of Birth,

(a) Search by Date of Birth (DOB) using the format MM/DD/YYYY.

(b) If more than 100 matches are found, you will need to further refine your search.

(4)   Email Address and/or Mobile Number,

(a) Email address search accepts up to 50 characters and must include the @  symbol.

           (b) Mobile number search is limited to United States mobile phones only.  Enter ten (10) digits without dashes.

 

Figure: DGI search chapter 33 screen

Click on the Search Chapter 33 button.

If records are found, they are displayed in a results table.  This indicates the beneficiary has potentially had a prior claim processed under chapter 33.

 

Figure: DGI search results example

 

If no results are found in DGI, a message displays reading No Results Found and a Search VADIR (VA/DoD Identity Repository) button appears.

 

Figure: DGI search results – No results example

 

 

 

 

II.2.D.3.b

Search VADIR

If no results are found when searching chapter 33, click on the Search VADIR button.

If records are found, they are displayed in a results table.  For beneficiaries that exist in VADIR, most personally identifying information is auto-populated in DGI. 

 

Figure: DGI VADIR search results example

If no results are found in either the chapter 33 or the VADIR search, the Veterans Claims Examiner (VCE) selects one of two (2) options:

(1)  New Chapter 33 Search to run a new search, or

(2)  Create New Chapter 33 Record, which navigates the VCE to the Claimant Biography screen (BIO) to create a new chapter 33 record in DGI. 

Figure: DGI VADIR search results – No results example

 

 

 

 

II.2.D.3.c

Wildcard Search

A wildcard search is available when a processor does not have complete information.

  For example: A poor-quality copy of the claim document was received where a name or SSN is only partially legible.

 Wildcard searches require the placement of a percent sign (%) at the end of the entered data.  Wildcard searches can be used in either the SSN or File # or the Last Name fields of the Search Chapter 33 screen.

  For example: A wildcard search using a partial SSN of 444 would be entered as “444%.”

If no matches are found using the wildcard search, the VCE will need to manually create a new chapter 33 record in DGI by clicking on the Create New Chapter 33 Record button.

 

 

 

 

II.2.D.3.d

Record Locked

If a record exists and is currently opened by another DGI user, a Record Locked message displays.  The VCE cannot access the locked record unless the other user either:

(1)  Releases the lock by navigating out of the record,

(2)  Logs out of DGI, or

(3)  Is inactive for 15 minutes.

II.2.D.4.a

General

The BIO screen for newly created DGI records is auto-populated if a claimant’s record is available in the VADIR.  The VCE must compare the populated information with the beneficiary’s application to ensure accuracy.

If there is no record in VADIR, the VCE must manually enter the required biographical information to create the new record in DGI.

For DGI records that

(1)  Had an original work product authorized prior to the conversion to eMPWR, and

(2)  Had no eMPWR Conversion work product, and

(3)  Had no work product authorized on/after July 8, 2024,

the BIO screen displays the information most recently entered into DGI.

For DGI records that already have had an eMPWR work product already authorized, the information that displays on the BIO screen will show the most up-to-date information from the VA systems of record for name, contact information, payment address information, and other biographical information.

VCEs should note that the information displayed on the BIO page may not always match the most recent information captured in The Image Management System (TIMS) when that information was received by a different VA line of business. If a VCE is uncertain if the information in TIMS is newer than the information in DGI, using the Corporate Inquiries option of the Share system allows for users to view effective dates for biographical information to be compared with documentation on file with EDU.

If the beneficiary has a fiduciary on file with the VA, a warning banner will display at the top of the BIO screen reading “Fiduciary Involved. BIO Changes must go thru the Fiduciary Hub.”

 

 

 

 

II.2.D.4.b

Claimant section

This section of the BIO screen contains the following information:

(1)  SSN field: The social security number is auto-populated from the search screen and cannot be edited.

(a)  If the SSN is incorrect, follow local procedures to submit a Service Now (SNOW) ticket to have the SSN corrected in the DGI system.

(b)  If the beneficiary does not have an SSN, a Temporary Identification Number (TIN) beginning with an 8, or Foreign Identification Number (FIN) beginning with a 9, is required in lieu of an SSN. If the FIN or TIN is not readily available, contact DGIB Policy & Procedures prior to record creation. In an encrypted email, provide the first, middle (if available), and last name, plus the date of birth of the beneficiary with no SSN, plus the first, middle (if available), and last name, date of birth, and SSN of the entitled Veteran. A response will be provided with further instructions, including the FIN or TIN if available.

(2)  Station field: User information that is auto-populated from the login screen for new Chapter 33 record creation.  For supplemental claims, this will initially display the most recently saved station. This field can be adjusted as necessary.

(3)  File Number field:  This is a required field and must contain eight or nine digits. An 8-digit number will have a "C" added to indicate a file number, while a 9-digit number indicates an SSN.

(a)  This field cannot be edited after a record has been created for a beneficiary.  If the file number is incorrect, follow local procedures to submit a SNOW ticket to have the file number corrected in the DGI system.

(b)  If the beneficiary does not have an SSN or file number, refer to the instructions in section 1(b) for beneficiaries with no SSN.

(4)  Fry (Veteran LOD) Date field: This field is auto-populated during the processing of a Fry claim based on the date of death entered for the Veteran during work product creation. It cannot be edited in the BIO screen for the beneficiary.  For in-progress original claims, this field remains blank.

(5)  POA Code field: The Power of Attorney (POA) Code field is used if the beneficiary has a power of attorney form on file or if one is indicated in VADIR. This field is required only if the beneficiary’s file indicates a POA. The processor can type in the name of a POA or select the POA from the dropdown.

(a)  A POA Name field displays if code 66 (Attorney-Exclusive Contact Requested) or 99 (Attorney-Exclusive Contact Not Requested) are selected.  In this scenario, the POA name is required.

(b)  If the POA from the Veterans Information System (VIS) is not listed in DGI, the VCE must select Stop Automation and edit all letters with the appropriate POA and print following local procedures.  See Automation Section for details.

(6)  6X Date field: If the six-credit hour exclusion (6X) was previously granted under chapter 30, chapter 32, chapter 35, chapter 1606, or chapter 1607, the VCE is required to enter that date in the 6X Date field. 

Previously used 6X for non-chapter 33 beneficiaries can be found in Benefits Manager, either under the Other section of the Beneficiary Information tab or in the banner of a BDN Historical Reporting report.

Figure: Other section of Beneficiary Information tab example showing

Six-credit hour exclusion field 

Figure: BDN Historical Reporting report banner example showing 

Six-credit hour exclusion field.

Note: Entering the 6X date stops DGI from applying the 6X a second time.  The 6X Date field also can be modified in a supplemental work product, which may affect lifetime award calculations.  A warning message displays below the screen header if the 6X date is modified. 

(7)  Sensitivity field: The sensitivity field indicates the beneficiary’s level of sensitivity. For example, a senator or high-ranking military officer may have a high level of sensitivity, indicating their record may need to be accessed by a VCE with approved clearance.

(8)  VADIR field: This field indicates whether the beneficiary’s file is associated with VADIR or not.  This is auto-populated.

Note: In the claimant section of the BIO screen, the File Number and Sensitivity fields are required to be filled in by the processor to continue processing the claim.  All other fields are either system-generated or are only required when applicable.

References: See M22-4, Part IV, 11.05 - 11.10 for details on the six-credit hour exclusion.

See M22-4, Part III, 1.11 for more information on sensitivity levels.

Figure: DGI BIO screen, claimant section example

 

 

 

 

II.2.D.4.c

Personal section

The personal section of the BIO screen displays name, gender, and date of birth information. This information is stored by VA in the Master Person Index (MPI).  When MPI is updated, the new information will display in DGI.  If EDU receives information that would warrant an MPI change for an existing beneficiary, follow local procedures to submit a SNOW ticket along with the appropriate documentation to have the information updated.

The appropriate documentation varies depending on the specific scenario as detailed below.

(1)   Correction of incorrect MPI information

Provide at least one un-expired primary identification document that displays the trait being corrected.

A secondary identification document might be needed, but is not sufficient for an MPI change.

Primary identification

Secondary identification

State-issued driver's license with current name, photo and DOB

Birth certificate

Passport with current name, photo and DOB

Social security card

Federal, state, or local government-issued photo ID containing current name and DOB

(2)   Record amendments – Name change

An amendment request to change a name requires that the individual submit the request in writing along with an un-expired government issued photo ID with DOB that displays the requested name.

Note: Marriage licenses or certificates are not sufficient documents for a name change, as not all people who apply for a marriage license or marry actually change their name. 

Identification

State-issued driver's license with new name, photo and DOB

Passport with new name, photo and DOB

Federal, state, or local government-issued photo ID containing new name and DOB

Note: Any changes to name, gender, or date of birth in DGI will be retained in all future EDU claims unless a newer change in made in MPI.  Changes made in DGI by EDU processors will NOT update MPI without a SNOW ticket submission.

(3)   Record amendments – SSN change

An amendment request to change the SSN requires that the individual submit the request in writing along with one form of un-expired primary identification and one form of secondary identification that displays the current SSN.  The primary identification verifies the person’s identity and the secondary identification verifies the updated SSN.

Primary identification

Secondary identification

State-issued driver's license with current name, photo and DOB

Social security card

Letter from SSA with updated SSN

Federal, state, or local government-issued photo ID containing current name and DOB

II.2.D.4.d

Additional Information section

This section of the BIO screen contains the following information:

Date of Death: This field is used to enter the date of death of the beneficiary when one is reported or found.  VCEs can enter or modify a date of death here, as needed. 

Note: For beneficiaries with a date of death in their BDN master record at the time of conversion to eMPWR, the date of death will be auto-populated in DGI.

After the transition to eMPWR, date of death will be pulled from MPI for inclusion in DGI without VCE action. When MPI is updated with a date of death, the new information will display in DGI.  If EDU receives information that would warrant an MPI change for an existing beneficiary, follow local procedures to submit a SNOW ticket along with the appropriate documentation (i.e., death notice) to have the information updated.

If the VCE is processing a first notice of death (FNOD) and no date of death is in MPI, the VCE must enter the date of death in MM/DD/YYYY format into the DGI system.  When formal documentation of death is received, local procedures to submit a SNOW ticket are followed to update the MPI record with the appropriate date of death. 

When an enrollment period is entered on the Enrollment screen of DGI and a date of death is present on the BIO screen, then a warning message displays directing the VCE to verify the date of death.

II.2.D.4.e

Contact Information section

The contact information section of the BIO screen displays phone number, email address, and mailing address information.  This information is stored by VA in the VA Profile system.  When updated by another VA business line, the new information will display in DGI. Similarly, when this information is updated by an EDU processor, then the beneficiary’s information in updated in VA Profile.

If a beneficiary already has contact information stored in VA Profile at the time of initial file creation in DGI, the contact information will be pulled in from VA Profile.

Education users cannot set up or edit a contact address for a beneficiary with a fiduciary.  If a change to an existing fiduciary is required, the user needs to contact the Fiduciary Hub to request an update.  See (I) 2022 14 September: Procedural Advisory: Changing Contact or Direct Deposit Information When a Beneficiary has a Fiduciary Assigned for full details on contacting the Fiduciary Hub.

Note: In order to protect interruptions in medical services, VA Profile will not overwrite mailing addresses with active prescriptions with changes made by EDU processors.  Any changes made by EDU will only reflect in Veterans Benefits Administration (VBA) systems, not VA-wide.  This is a systemic protection and has no impact on how VCEs process EDU claims.

Note: Addresses should no longer be adjusted based solely on formatting issues unless we receive returned mail due to the formatting of the address. If a new address is being entered manually, the address standards here must be followed.

If the address is not found to be valid by the United States Postal Service (USPS), any associated work product will fail authorization and development to the beneficiary will be needed to find a valid mailing address.  See M22-4, Part II, 2.D.14 for more details on authorization errors.

See the glossary of Appendices for further details on VA Profile.

II.2.D.4.f

Payment Address section

Prior to July 8, 2024, the Payment Address for beneficiaries was stored in BDN as the system of record for payment generation.  With the transition to eMPWR, the official system of record for payment addresses has become the Corporate Database (CorpDB) which is shared across VBA.  Education processors do not have direct access to CorpDB but can view information from the CorpDB in SHARE and update and view payment addresses in DGI. 

See the Glossary of Appendices for further details on CorpDB. 

Beneficiaries may only use one payment address for all their VA benefits.  The implications for this, which were not a factor with BDN, will be detailed in this section.

For some benefits processed in DGI: chapter 33 (Self, TOE, or Fry) and STEM, DGI is the route by which EDU updates a beneficiary’s payment address in CorpDB.  VET TEC, which is also processed in DGI, will continue to update their payment address directly in Share until integration with eMPWR is complete.

Beneficiaries’ payment addresses from CorpDB will NOT display in DGI until either (1) an eMPWR conversion work product is authorized, or (2) the first work product on or after July 8, 2024 is authorized.  The active payment address and its effective date do display for comparison purposes.  This is because the beneficiary does not have an education record in CorpDB until that initial work product is authorized.

Any changes made by EDU to a beneficiary’s payment address will change their payment address for all VA business lines.  Be sure the most recent payment address is always used.

Adding or updating a payment address in DGI.

(1)  Confirm that making a change is appropriate.  See the scenarios below to determine what is appropriate.

(2)  Select the appropriate type of payment address.

(a)   Select: This is the default option.  If it is not appropriate to add a payment address, no adjustment is needed, and the field should display Select.  An example of a situation to not enter a payment address is the creation of claimant biographies for TOE dependents (other than the applicant) upon initial veteran setup when other dependents have not applied, and thus, have not provided a payment address.

(b)  Direct Deposit: Select if the beneficiary indicates they are using Electronic Funds Transfer (EFT) to receive their payments.

•   Account Type:  The user is required to select either Checking or Savings.

•   Routing Number: This field is required for EFT payment addresses.  This field only accepts numeric characters, must be 9 digits, and a valid US routing number. If an invalid routing number is entered, an error message will display.

•   Account Number: This field is required for EFT payment addresses.  This field accepts only numeric characters and must be between 5 and 17 digits.

Note: Only domestic United States EFT addresses can be used for EDU direct deposit payments.  No foreign EFT payment addresses can be entered into DGI.

Note: If a beneficiary is using an International Direct Deposit (IDD) payment address for a non-EDU benefit, they can continue to do so.  In this case, all EDU benefits will be paid via a paper check sent to their mailing address until such a time that EDU can handle IDD payment addresses.  If EDU updates the payment address in DGI, all payment addresses, including the IDD, will be updated to the new address.  If the beneficiary wants to continue using their IDD to receive non-EDU benefits, EDU users should not make any changes to the payment address in DGI.

 

 Figure: DGI Payment Address – Direct Deposit example

 (c)   Physical Address: Select if the beneficiary does not provide EFT. When the following message displays, Does the mailing address match the payment address? the processor selects either Yes or No, as appropriate.

•   When the VCE selects Yes, the existing mailing address on the Bio screen automatically populates as the payment address.

•   When the VCE selects No, then the user is required to enter a physical address for a paper check to be sent.  This could be a different address than the mailing address where the letter is sent. 

Reference: See the Contact Information section for address entry rules.  

(d)   Fiduciary: Education users cannot set up or edit a fiduciary payment address.  If a change to an existing fiduciary is required, the user needs to contact the Fiduciary Hub to request an update.  See (I) 2022 14 September: Procedural Advisory: Changing Contact or Direct Deposit Information When a Beneficiary has a Fiduciary Assigned for full details on contacting the Fiduciary Hub. 

    (3)   To update payment addresses, determine which of the following scenarios is appropriate and follow the directions:

(a)   Original claim in DGI.

•    Add payment address as indicated on the application if the date of receipt is more recent than the date showing in the Last Updated By field.

•    Open an original work product and process using standard procedures, including sending for second signature authorization.

•    Upon authorization, DGI compares the payment address against CorpDB.  If a payment address has been added to CorpDB in the interim AND it does not match the payment address on the BIO screen, the Senior VCE is prompted to review which payment address is more recent by comparing Last Updated By on the BIO screen with the date of the most recent payment address in The Image Management System (TIMS) file.  The Senior VCE selects the more recent address.  If the payment address from CorpDB returns as a fiduciary address that differs from the payment address from the claim, contact the Fiduciary Hub to request an update.  See (I) 2022 14 September: Procedural Advisory: Changing Contact or Direct Deposit Information When a Beneficiary has a Fiduciary Assigned for full details on contacting the Fiduciary Hub.

Note:  Users should enter the payment address provided on any original application when reviewing the BIO screen since eligibility cannot be determined with certainty until the work product is fully processed.  DGI is designed to automatically remove payment addresses for beneficiaries whose claim is denied.  As such, upon authorization of a chapter 33 denial, the payment address will be removed automatically and will display as blank.

 

 Figure: DGI Payment Address – Original claims example

(b)   Supplemental claim in DGI with no conversion work product and no authorized work product after July 8, 2024.

•    Compare the date of the most recent payment address on file from the beneficiary’s file in TIMS with the date of the most recent CorpDB payment address shown in the Last Updated By field.

•    If appropriate, enter the more recent payment address on the BIO screen.

•    Select Next button on the BIO screen.

•    Process the claim using standard procedures, including sending for second signature authorization, if necessary.

•    Upon authorization, DGI compares the payment address against CorpDB.  If a payment address has been added to CorpDB in the interim AND it does not match the payment address on the BIO screen, the Senior VCE is prompted to review which payment address is more recent by comparing Last Updated By on the BIO screen with the date of the most recent payment address in The Image Management System (TIMS) file.  The Senior VCE selects the more recent address.  If the payment address from CorpDB returns as a fiduciary address that differs from the payment address from the claim, contact the Fiduciary Hub to request an update.  See (I) 2022 14 September: Procedural Advisory: Changing Contact or Direct Deposit Information When a Beneficiary has a Fiduciary Assigned for full details on contacting the Fiduciary Hub.

(c)   Supplemental claim in DGI with a payment address existing in CorpDB and an eMPWR conversion work product, an authorized work product on or after July 8, 2024, or both.

•    Compare Last Updated By on the BIO screen with the most recent payment address on file from the beneficiary’s file in TIMS.  The most recent payment address is to be used.

•    Update payment address on the BIO screen if a more recent one is found.

•    Select Save button on BIO screen.

•    Proceed using standard procedures.

(d)   Supplemental claim in DGI with a fiduciary payment address existing in CorpDB and an eMPWR conversion work product, an authorized work product on or after July 8, 2024, or both.

•    Compare Last Updated By on the BIO screen with the most recent payment address on file from the beneficiary’s file in TIMS.  The most recent payment address is to be used.

•    Contact the fiduciary hub to update, as appropriate. See (I) 2022 14 September: Procedural Advisory: Changing Contact or Direct Deposit Information When a Beneficiary has a Fiduciary Assigned for full details on contacting the Fiduciary Hub.

•    Hold claim until fiduciary hub confirms address has been updated.

•    Proceed using standard procedures.

If a beneficiary has a payment address in CorpDB (from an earlier Education claim or from another business line), then the most recent address displays in the Payment Address section of the BIO screen.

Last updated By field

The Last updated by field displays the source and date of the most recent change to the beneficiary’s payment address.  Possible sources of change include the following.

•    DGI – If the change is made in DGI, the name of the user who made the change displays.

•    MyEducationBenefits (MEB) – only the system of the source (MEB) displays.

•    VA.gov – only the system of the source (VA.gov) displays.

•    CorpDB – only the system of the source (CorpDB) displays.

VCEs must review this field before making any changes to payment address to verify the address they have is more recent than the one on file.

Payment Address History button

Under the last updated by field, a Payment Address History button is available.  Selecting this button opens a window that displays a history of payment address changes.  This window shows the current and all prior payment address information, the user/system that made the change, and the date of the change assisting with payment tracking and audit activities.

Note: Due to integration issues with EDU partner systems, it is possible that some duplicate lines may display for physical payment addresses at foreign addresses and may be disregarded. 

02X Fraud blocker

To protect beneficiaries from fraud, eMPWR has a method to prevent fraudulent mailing and/or payment address updates.  Only users of the VBA Incident Team at the Salt Lake City Regional Office (station 341) can set or modify the fraud blocker.

All attempts to change an address through other applications (e.g., Share, DGI, etc.) will return a message to the VCE indicating the change is not allowed and will be elevated for further review. 

 The attempted address change will then be written to the new eMPWR Elevated Change of Address Request Report which is managed by the VBA Incident Team. 

If it is determined that the change is legitimate, an authorized user at the SLC RO will change the address.

II.2.D.4.i

Suspended File section

Since BDN was a processing system as well as a payment system, it was able to stop payments from going out in certain circumstances to prevent certain erroneous payments.  eMPWR does not have this ability since it is only a payment system.  As such, a Suspense section has been added to the DGI BIO screen to allow this to happen manually.  Placing a beneficiary’s file in suspense  in DGI will result in certain payments not being released until the suspense is resolved.  Examples of reasons for a file to be suspended include the return of payments as undeliverable, report of a date of death, or a potential administrative error.

 The “Suspended Claimants Report” provides a list of all suspended claims, age, and reason, broken down by Regional Processing Office (RPO).

(1)  Putting a file into suspense

When a VCE identifies a situation that requires payments to be suspended, the following steps are taken to suspend the file.

(a)   Select the Suspend Payment Transactions box on the BIO screen.

(b)   Select a reason in the resulting Reason for Suspense dropdown menu.

(2)   Working a suspended file

While, in most cases, VCEs should resolve a suspense issue before finishing a claim, it is possible that a file will stay in suspense for a longer period of time and processing will be required while the file is in suspense.  DGI allows users to create work products for files that are in suspense.

 The VCE must review the BIO screen before processing any claim and will see the Suspend Payment Transactions box checked along with the reason for suspense.  A second box appears reading Verified Suspense Status.  In order to authorize the claim, the VCE must check this box after verifying the validity of the suspense reason.

 If the Verified Suspense Status box is unchecked, upon reaching the Work Product Summary (WPS) screen and attempting authorization, an authorization error message displays instructing the authorizer to review the Suspense status on the BIO screen. 

(3)   Removing a suspense

When the VCE resolves the suspense issue, DGI allows the VCE to indicate the resolution and will release the suspended payments.  To do so, the VCE will take the following steps.

(a)   De-select the Suspend Payment Transactions box.

•   If the reason is Date of Death Found, the VCE must also remove the date of death from the BIO screen before suspense is released.

(b)   A confirmation message appears to inform the VCE that this action will result in payments being released.

(c)   All BTs in suspended status will be reset to the appropriate status of either Successful or Scheduled.  See the Payment Actions section of the Net Awards Topic 15: Payment Instructions Screen for details of M22-4, Part II, 2.D.15.

Note: Once a suspense has been released, the only remaining indicator in DGI that the claim had been suspended in an “Unchecked by” field identifying the user who removed the suspense.  Further details can be gathered using the Claimants in Suspense Report.

Important: Be sure all suspended payments are due before removing the suspense.  If there are improper payments reflected, be sure to process a work product while the file is suspended to correct the payments.

Note: Call center users can only modify the payment status if the suspense reason is Beneficiary payment undeliverable and the user has validated the payment information.

(4)   Additional information on death claims

In addition to adding a suspense for date of death, several other updates have been made to allow DGI to better handle death claims upon transition to eMPWR.

In the situation when payment has been issued prior to date of death being reported and entered, the following actions occur.

    (a)  Vendor debts (75B) may be generated (for tuition and fees or Yellow Ribbon at an Institution of Higher Learning (IHL) or Non-College Degree (NCD) institution).

•   Date of death is during the term.  A letter must be created by the VCE explaining that a refund may be due to VA.  No 75B debt is created unless the school returns a portion (or all) of the monies paid.  The debt is only created in the amount of the returned funds.  The actual debt creation is done by the Support Services Division (SSD).

•    Date of death is on or before the first day of the term.  The 75B debt is generated and a debt letter is sent to the school, but the debt is NOT automatically sent to eMPWR.  If appropriate, the VCE follows local procedures to request SSD manually create the debt.

Note: VCEs should avoid creating a work product after entering a date of death without a corresponding Amendment of Enrollment (1999B) document from the school.  DGI may attempt to create a 75B debt with a date of death entered and without an amendment from the school.  If this occurs, VCEs must develop to the school for the appropriate certification before processing the claim.

(b)  Housing – a debt is calculated by DGI but will not be established against the beneficiary or their next of kin regardless of the timing of the death.

(c)  Books and supplies – a debt is calculated by DGI but will not be established against the beneficiary or their next of kin regardless of the timing of the death.

(d)  Chapter 30 or chapter 1606 kicker – a debt is calculated by DGI but will not be established against the beneficiary or their next of kin regardless of the timing of the death.

If an enrollment period is in progress, and a date of death is added to the file, a warning message will display on the WPS page indicating that a date of death is on file.  The enrollment may still be entered, but nothing will be paid after the date of death.  Any payments due will be prorated based on date of death.

II.2.D.4.j

Automation section

The Automation section is used whenever a processing concern is identified that could result in erroneous payments being released if a subsequent claim was authorized.

 This section of the BIO screen contains the following functionality:

Stop Automation box – select this box to cause any claims received for this beneficiary to not automate.

Checked by field – this field displays the VCE who initially checked the Stop Automation box and the date of the action.

Verify Continue Stop Automation box – this box must be checked before any subsequent work product can be authorized to confirm the continued validity of the stop.

•    If the Verify Continue Stop Automation box is unchecked, upon reaching the WPS and attempting authorization, an authorization error message displays requiring the authorizer to review the stop automation section of the BIO screen

Note field – this is a free text field required any time the Stop Automation box is checked.  It provides the reason for the stop.

Approved reasons for using the stop automation box are the following:

•   Chapter 31

•   Dual source TOE

•   OOS

•   Incarcerated Veteran

•   Kicker entitlement

•   Foreign school

All other stop automation reasons have been resolved by previous system updates.  If a VCE locates a beneficiary with the stop automation box selected for a reason other than the 6 above, they must remove the stop automation selection.

In order to remove a stop automation, the VCE must unselect the stop automation checkbox, enter an updated note, and select Next.
