**SUBCHAPTER I.  OVERVIEW OF NOTICES OF EXCEPTION PROCESSING**

**8.01 GENERAL**

---

This chapter contains Target processing and procedures that do not readily belong in other chapters of M22-4. 

**a.** [Subchapter I](#SI) deals with general Notices of Exception processing.

**b.** [Subchapter II](#SII) deals with Target commands not covered elsewhere.

**8.02  NOTICES OF EXCEPTION PROCESSING - GENERAL**

---

NOE (Notices of Exception) result from TCS (Target Central System) processing.  At times, Target processing may result in rejection of an action or in generation of an information message.  If Target rejects an action or sends a message, TCS sets up an NOE message.  Target sends the messages to the BDNRS (Benefits Delivery Network Regional Systems) for further processing and retrieval by the regional offices.  Regional office personnel obtain a list of NOE messages by using the NOES (Notice Of Exception Subsystem) command.

**8.03  ACCESS TO THE NOES COMMAND**

---

The Adjudication Officer designates certain individuals to have access to the NOES command.  An operator must request a separate listing for each education benefit system.  For example, an operator would have to submit a request for a chapter 34/35 list and a second request for a chapter 32 list.  NOE listings within a given education benefit system (except for chapter 32) contain all NOE messages for the entire field station (digits 00-99).  (NOE listings for the chapter 32 system contain all NOE messages within a maximum of 20 digits in each request.)  The NOES command has limited access to prevent the unnecessary printing of duplicate listings.  Generally, field stations should assign one individual to obtain the NOES listings; several other individuals should also receive training as alternates and have access to the command. 

**8.04  TYPES OF NOE MESSAGES**

---

There are three categories of NOE messages:  REJECT, ACTION, and INFO messages.  Each message requires regional office attention.  The category of the NOE message determines the type of action required and the priority assigned.

**a.  REJECT Messages (Chs. 32 and 35)**.  REJECT messages should have the highest priority.  They indicate that TCS did not successfully process a transaction.  Regional offices should carefully review claims involving REJECT messages.  These messages often require submission of corrected action to the TCS.

**b.  ACTION Messages**.  Regional offices should carefully review claims involving ACTION messages.  These messages often direct attention to suspended awards, required certifications, or changes in eligibility.  ACTION messages do not usually reflect an error by a VA operator.

**c.  INFO Messages**.  These messages should have the third highest priority.  They identify unusual TCS processing circumstances.  Regional offices should carefully review claims involving INFO messages and take any necessary action.  These messages usually contain information for documentation purposes.  Adjudicators may often file INFO messages in the claims folder without further action.

**8.05  REQUESTING NOE MESSAGES**

---

(1)  Make the following entries on the Ready screen to obtain a NOTICE OF EXCEPTION MENU SELECTION SCREEN (S12 for chapters 30, 32, and 106, S11 for chapter 35):

(a) The command NOES.

(b) Operator's password.

(c) Benefit.  If CH35, leave blank.

(2)  Make the following entries on the NOE menu selection screen to have Target print NOE messages:

(a) Specify any range of digits (00 thru 99) for NOE messages.

(b) Put an X under the date or cycle by desired NOE messages.

(c) Press the ENTER key.

**a.  NOE Fields for Chapter 30, 32, and 106 Systems**

1.  PROCESSING DATE.  This field shows the date of NOE occurrence (month and day).

2.  STATION NUMBER.  This field shows the regional office number of the station processing the claim or having jurisdiction over the claim.

3.  FILE NUMBER.  This field shows the claimant's file number.  The social security number shows for chapter 32 claimants.

4.  PAYEE NUMBER.  This field shows payee 00.

5.  STUB NAME.  This field shows the payee's stub name if transaction processing generates the NOE.  This field shows the master record stub name if recurring payment processing or bank processing generates the NOE.

6.  REJECT MESSAGE CODE.  This field shows the NOE code number.

7.  TRANSACTION CODE.  This field identifies the transaction that caused the NOE.

8.  UNASSOCIATED NUMBER.  (ch. 32 only)  This field is blank or shows a file number.  Route the message to the finance activity for review if a file number shows in this field.

9.  SEQUENCE NUMBER.  (ch. 32 only)  This field is blank or shows two digits.  Route the message to the finance activity for review if two digits show in this field.

10.  BATCH NUMBER.  (ch. 32 only)  This field is blank or shows four digits.  Route the message to the finance activity for review if four digits show in this field.

11.  AMOUNT.  (ch. 32 only)  This field is blank or shows a dollar amount representing unassociated money in the record.  Route the message to the finance activity for review if unassociated money shows in this field.

12.  REJECT MESSAGE LEGEND.  This field has an explanation of the NOE code number.

**b.  NOE Fields for Chapter 35 System**

1.  CYCLE NUMBER.  This field shows the cycle of NOE occurrence.

2.  BATCH NUMBER.  This field shows four digits.  The first digit is the last digit of the year.  The last three digits are the Julian date representing the day TCS assembled this transaction with other transactions and stored it for processing.  For example, Batch 9018 indicates that the Hines BDC (Benefit Delivery Center) assembled the transaction on January 18, 1989.  TCS processed transactions in this batch during Cycle 0, February 1989, on January 20, 1989.

3.  STATION NUMBER.  See subparagraph a above.

4.  BATCH DATE.  This field shows the calendar date of the day TCS assembled this transaction with other transactions (at the Hines BDC) and stored it for processing by the TCS.

5.  TRANSACTION CODE AND MODIFIER.  This field identifies the transaction that caused the NOE. 

6.  FILE NUMBER.  This field shows the claimant's file number.

7.  PAYEE NUMBER.  This field shows the payee's number.

8.  STUB NAME.  This field shows the payee's stub name if transaction processing generates the NOE.

9.  REJECT MESSAGE CODE.  This field shows the NOE code number.

10.  REJECT MESSAGE LEGEND.  This field has an explanation of the NOE code number.

**8.07  DISPOSITION OF NOE MESSAGES**

---

Target prints NOE messages in terminal-digit order.  Regional office personnel separate NOE messages and send them to the appropriate claims unit for association with the appropriate folder and award action, as required.  Station personnel dispose of NOES messages, when no longer required, in accordance with Records Control Schedule VB-1, Part I, item No 08-041.000.

**SUBCHAPTER II.  MISCELLANEOUS TARGET COMMANDS**

**8.08  DUPC**

---

DUPC (DUPlicate Record Consolidation) is a BIRLS (Beneficiary Identification & Records Locator Subsystem) command which consolidates two BIRLS records, based on an "old file number" (cancelled file number) and "new file number" (retained file number).  (See M21-1, pt. II, par. 3.16 for DUPC procedures.)  These BIRLS records may or may not have a Hines benefit master record.  DUPC processing generates a change of file number transaction (26) for any chapter 30 or 34 record existing under the "old file number" (cancelled number).  DUPC cancels the Hines benefit master record under the cancelled file number.  For chapter 34, an operator uses the CORR (correction) command to update the retained record to reflect the entitlement used under the cancelled file number.  For DUPC, the operator must have physical control of both folders.  BIRLS must also show the operator's regional office has jurisdiction of both folders.  See [paragraph 8.09](#809) for the changing of a file number.  This processing is used when BIRLS contains the correct file number with a Hines master record under a different number.

**a.  Retention Criteria**.  See M23-1, part I, paragraph 6.05 for information as to how to determine which record to retain.

**b.  Chapter 30 DUPC Processing**.  See [M22-4 Part 5: Chapter 2 - MGIB BDN Processing](https://eg5813aprl-int-nlb.egainstz.cloud/system/ws/v11/ss/article/554400000076230?portalId=554400000001018&usertype=customer&\$lang=en-us&)    

**8.09  CNUM**

---

The operator uses the CNUM (Change of File NUMber) command to change the file number for a benefit master record to a different number.  An operator can currently use CNUM to change records in the chapter 34/35 and chapter 30 systems.  An operator uses this command when there is only one Hines master record under a number different than the BIRLS reference number.  See [paragraph 8.08](#808) for DUPC procedures when two BIRLS records exist (with or without a Hines master record under each BIRLS number). 

**a.  CNUM Procedures**

(1)  The required Ready screen entries are: 

(a)  The command CADJ,

(b)  Operator's password,

(c)  The file number. 

(d)  Benefit type.    

(e)  For chapter 35, also enter the payee number.  For chapters 30 and 34, the system assumes payee 00.

(2)  If Target does not locate a record with the file and payee number, CNUM processing terminates and the Ready screen shows "NO MASTER RECORD IN FILE.

(3)  If Target locates an abbreviated chapter 34 master record, CNUM processing terminates and the Ready screen shows "CNUM CANNOT BE PROCESSED-REACTIVATE RECORD".

*NOTE: See*[Part 7: Chapter 2 - BDN Processing](https://eg5813aprl-int-nlb.egainstz.cloud/system/ws/v11/ss/article/554400000073682?portalId=554400000001018&usertype=customer&\$lang=en-us&)*for instructions on activation procedures for chapter 34 records.*

*(4)  If Target locates a record with the file and payee number, CNUM processing proceeds.  Target shows the M21 screen.  Type the correct number over the incorrect number and press ENTER.*

*(5)  BIRLS checks for a record under the new number containing the correct name.  If BIRLS does not have such a record, Target terminates CNUM processing.  The Ready screen shows the message "RECORD BELONGS TO (SHOWS NAME OF OTHER INDIVIDUAL)-DO BUPD OR CNAM" or the message "NO BIRLS RECORD UNDER NEW FILE NUMBER-DO CEST".*

*(6)  If a master record already exists for the new file number and payee number, CNUM processing terminates.  The Ready screen shows "RECORD EXISTS FOR NEW FILE NUMBER."*

*(7)  When CNUM processing is complete, the Ready screen shows "PROCESSING COMPLETE."  Target forwards a 26 transaction to the Hines BDC.*

*(8)  When CNUM processing is complete for a chapter 34 record containing a chapter 30 indicator, Target sends VA Form 22-8270a, Education Master Record - Audit Write Out to the regional office that processed the CNUM transaction with a 903 message showing "TR 26 PROC AGAINST POSSBL 34/30 M/R - REVIEW."*

*NOTE:  Transaction 26 takes two input cycles to update a number in the chapter 34/35 system.*

**b.  Chapter 30 System**.  Only the RPOs (regional processing offices) can use the CNUM command to change chapter 30 records.  (See [PART 02, COMPUTER PROCESSING. CHAPTER 2 - TRANSACTION CODES](https://eg5813aprl-int-nlb.egainstz.cloud/system/ws/v11/ss/article/554400000076474?portalId=554400000001018&usertype=customer&\$lang=en-us&) )

*NOTE:  If an operator at a regional office encounters a record involving potential chapter 30 record CNUM adjustment (e.g. while reviewing a claims folder with a C&P record), he/she should bring this record to the station chapter 30 coordinator.  Before taking any CNUM action, that individual should coordinate with his/her counterpart at the chapter 30 RPO.*

***c.  Alternatives to CNUM***

(1)  Chapter 34/35 system.  For a limited number of cases, Target has two master records for the same claimant while BIRLS only has one record.  See [paragraph 8.10](#810) for necessary PURGE actions.  An operator cannot use the CNUM command in this situation.

(2)  Chapter 32/903 system.  CNUM processing is not available for the chapter 32/903 system.  For the chapter 32/903 system, Target should never have two benefit master records for the same claimant because a bank record is necessary for the proper establishment of a benefit record.  If an operator encounters such a case, he/she should carefully review all records to determine if both records belong to the same individual.  This will usually require development to the service departments that sent the bank records.  The operator should determine the correct number.  The chapter 32 coordinator should coordinate the consolidation action by calling the Education System Staff (224A) at [(202)  273-7168].  Alternately, this coordinator can direct a message through Wang Office to "MAIL VBA 22" for further delivery to the System Staff (224A).  He or she should explain the reason for the request and give all pertinent file numbers.  The consolidation action requires Hines BDC action.  The Education System Staff notifies the chapter 32 coordinator when Target allows appropriate award action.

(3)  Chapter 106 system.  An operator cannot use the CNUM command.  For the chapter 106 interim system, Target should never have two benefit master records for the same claimant.  If, after appropriate development, both records belong to the same individual, the regional office 106 coordinator should call the Education Procedures Staff (224B) at [(202) 273 -7181].  Alternately, this coordinator can direct a message through Wang Office to "MAIL VBA 22" for further delivery to the Procedures Staff (224B).  The Education Procedures Staff notifies the chapter 106 coordinator after purging the benefit master record under the incorrect number.

(4)  DOD Records (Chapter 30/106 systems).  An operator cannot use the CNUM command.  For these systems, Target should never have two DOD (Department of Defense) records for the same claimant under two different numbers.  A DOD record is necessary for the proper establishment of a chapter 30 or 106 benefit record.  (See  [Part 2: Chapter 4 - Master Record Inquiry (Chapter 30)](https://eg5813aprl-int-nlb.egainstz.cloud/system/ws/v11/ss/article/554400000077340?portalId=554400000001018&usertype=customer&\$lang=en-us&) for DOD records.)  DOD records can have multiple entries (one from each branch or component) under one number.  If, after appropriate development, both DOD records belong to the same individual, the regional office chapter 30 or 106 coordinator should request the service contact point to delete the incorrect record.  For chapter 106, see [Part 8: Chapter 8 - Miscellaneous](https://eg5813aprl-int-nlb.egainstz.cloud/system/ws/v11/ss/article/554400000075619?portalId=554400000001018&usertype=customer&\$lang=en-us&) for the contact points.  For chapter 30, see  [Part 5: Chapter 8 - Miscellaneous](https://eg5813aprl-int-nlb.egainstz.cloud/system/ws/v11/ss/article/554400000076326?portalId=554400000001018&usertype=customer&\$lang=en-us&)  for the contact points.  The contact point notifies the coordinator after deleting the DOD record under the incorrect number.

**8.10 PURG**

---

The PURG (PURGe) command (available only for use by selected Central Office personnel) is available to delete a chapter 34/35 or 106 master record from the Hines BDC data base.  The PURG command will be available in the future for chapter 30 records.  Processing of the PURG command generates a transaction (26P) to the chapter 34/35 record.  Processing of this transaction generates a master record writeout containing the status of the master record immediately prior to the deletion.  The master record is deleted from the master file.  For chapter 34 and 35, the master record must be other than type A or D, must not be an abbreviated master record, and must not contain an accounts receivable.  Regional offices desiring deletion of a master record must submit a request explaining the circumstances, in writing or through Wang Office (to "MAIL VBA 22") to the Director of the Education Service (22).  See [paragraph 8.09c(3)](#809c3) for 106 purge procedures.

**8.11 STAT**

---

The STAT (STATus) command allows employees to obtain information concerning the current status and availability (either "available" or "unavailable") of the files in the Target system.  The S94 screen shows the current status and availability of the C&P (Compensation and Pension) and chapter 34/35 master files.  The S95 screen shows the current status and availability of the REPS (Reinstated Entitlement Program for Survivors) and chapters 30, 31 and 32 files.  One system can be available for review or input with others not being available.  Also, even if a person reviews the STAT screen, determines that a particular system is available, and then immediately attempts to access that system, the system may be unavailable.  That system may be having difficulties that the STAT screen does not reflect.  Also, the Hines BDC could have been in the process of taking that system down when the employee was reviewing the STAT screen.  Target shows an appropriate message when a system is not available.

**a.**  To access a STAT screen, the operator must complete a Ready screen with STAT as the command.  (See [fig. 8.09](#f809) for Ready screen entries.)  Target does not require the employee's password.  With these entries, Target automatically shows the S94 screen.  To review the S95 screen, the operator must include "S95" in the screen number field.  (See [fig. 8.10](#f810) for the S94 screen and [fig. 8.11](#f811) for the S95 screen.)

**8.13  CORR**

---

An operator may use the CORR (CORRection) command to make limited education master record transaction 02 and 02V corrections directly on a Target inquiry screen for all benefits except chapter 106.  This is an easier method to make corrections than preparing amended awards.  Also, see the alternate means of input for specific chapter 35 02V transactions ([pt. VII, ch. 2](https://vbaw.vba.va.gov/bl/22/ref/m22-4/Part%20VII/ch02.htm)).  Also, for chapter 30 CORR transactions, see part [M22-4 Part 5: Chapter 2 - MGIB BDN Processing](https://eg5813aprl-int-nlb.egainstz.cloud/system/ws/v11/ss/article/554400000076230?portalId=554400000001018&usertype=customer&\$lang=en-us&)  V, chapter 2 for detailed instructions.

**a. CORR Processing (Chs. 32/35)**

(1) An operator can make changes to the Education master record on the following screens:

(a) M21 - Education Master Record Status Screen;

(b) M23 - Education Master Record Eligibility and Entitlement Screen; and

(c) M24 - Education Master Record Miscellaneous Data Screen.

(2) Target has a limited number of correctable fields.  "Protected" fields are fields in which the operator cannot change the entry.  Target does not process changes, additions or deletions entered into these fields against the master record even though the screen appears to accept the entry.  Any field not shown in subparagraph c is a protected field.

(3) An operator must make a correction on only one of the screens if a particular field appears on more than one education inquiry screen.

(4) Corrections to the Hines master record do not affect BIRLS records.  If BIRLS corrections are necessary as a result of corrections to the Hines master record, follow current procedures for BIRLS update.

(5) Entries in Target correctable fields process in the next education pay cycle (or benefit daily processing for chapter 32).  This processing must occur before the corrections appear on the MINQ or CORR screens.

**b.  CORR Procedures (Chs. 32/35)**

(1) Ready Screen Entries

(a) An operator enters the command CORR, the operator's password, and the file number.  Also enter CH32 as the benefit if correcting a chapter 32 record.

(b) Optional entries include screen number, payee number and CH 35 as the benefit.

(c) Target shows the Payee Selection screen (SO2) for selection of the desired master record if entries on the Ready screen are insufficient to identify a specific record. 

(2) Master Record Corrections

(a) First determine if a field is correctable.  [Subparagraph c](#813c) lists the correctable fields by type of master record.

(b) If the field is correctable, erase the old data and enter the correct data if the field is correctable.  (On a chapter 35 record, an operator may not enter the data under CORR if a field is blank.)

(c) After making all the corrections on a screen, press the print key and request a printout of the corrected screen before pressing ENTER.  (Pressing ENTER sends the CORR transaction to Hines.)

(d) Sign the printout and file it in the claims folder.

(e) Target automatically makes the correction to the field on the other screen(s) when the operator makes a correction to a field that is correctable on more than one screen. 

**c.  Correctable Fields (Chs. 32/35)**.  The fields shown on the charts below are the only ones correctable on Target using the CORR command.  If a field is not shown below, Target will not process a CORR transaction to that field.  (NOTE: Target CORR transaction edits ensure that the entered information is correct.  The data shown on the charts below contain only the minimum basic entries for CORR processing.)

**8.14  FINANCE COMMANDS**

---

The following [BDN] commands are used by the Finance activity.  (For further information or details, see the appropriate finance manual.)

**a.  COLL (Collections and Returned Checks)**

COLL is used to process cash collections and checks returned for reason of insufficient funds which were previously applied as cash collections.

**b.  FAUT (Fiscal Authorization)**

FAUT is used to approve, disapprove, or bypass a pending fiscal authorization.

**c.  FIST ([Fiscal Transaction])**

The [FIST command is used] to release payments (i.e., special payments, one-time payments, work-study, tutorial payments, and to establish or adjust accounts receivable).

**d.  FOES (Financial Message System)**

The [FOES (Fiscal Notice of Exception System) command is used] in conjunction with fiscal output from BDN.  The FOES command operates similarly to the NOES command, but is limited to the Finance activity.

**e.  FRPT (Fiscal Transaction Report)**

The [FRPT command is used to obtain pending and authorized fiscal] transactions.

**f.  PENF (Pending Fiscal Transactions)**

The [PENF command is used to obtain pending fiscal transactions (similarly to the WIPP, Work in Process] command).

**g.  [RFND (Refund)**

The Refund command is used to both disenroll a veteran or serviceperson from the VEAP program and process a refund of that individual's unused VEAP contributions.  (The Finance activity uses the RFND command on the D11 Disenrollment Refund Benefit Chapter 32 screen.)

**h.  RAUT (Refund Authorization)**

The Refund Authorization command is used to approve or disapprove both the disenrollment of a veteran or serviceperson from the VEAP program and the release of a refund of unused VEAP contributions.  (The Finance activity uses the RAUT command on the D13 Refund Authorization Benefit Chapter 32 screen.)

**i.  UNAS (Unassociated Accounts)**

The UNAS command is used to view and clear unassociated chapter 30 funds which generally represent funds due VA payees.]

**8.15  LIAISON AND COMPLIANCE COMMANDS**

---

The following [BDN] commands are used by the Education section in VSD (Veterans Services Division).  (For further information or details, see [circular 22-83-12, Veterans Services Division Education Activities End Products on the [BDN] System.  This information is scheduled to be incorporated into pt. IX.)]

**a.  WEST (Work Establishment)**.  This command is used to establish a pending end product for a compliance or liaison issue.

**b.  WCAN (Work Cancellation)**.  This command is used to cancel a previously established pending issue.  No work measurement credit is generated by use of the WCAN procedure.

**c.  WCLR (Work Cleared)**.  This command is used to clear an existing pending issue or to take end product credit for an issue for which a pending issue was not required.

**d.  WINQ (Work Inquiry)**.  This command is used to review the status of pending issues for monitoring and appropriate follow-up commands.

---

The following [BDN] commands are used by Adjudication and VSD managers.  [ ]

**a.  WIPP (Work in Process)**.  This command is used to access the WIPP Subsystem to monitor pending issues.  Additional information is available on overall processing time and maturing issues.  [(For further information, see [M21-1, pt. V, ch. 22.)]

**b.  QCRE (Quality Control and Review)**.  This command allows the user to obtain weekly listings of completed end products for purposes of quality review.  [(For further information, see M21-4,  ch. 2.)]
