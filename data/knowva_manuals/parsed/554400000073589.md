**5.01  CHAPTER 32 NOE PROCESSING - GENERAL**

---

**a.**  After chapter 32 awards or other transactions are processed, they are sent to the TCS (BDN Central System) for further editing and storage.  TCS generates chapter 32 reject, action, and informational NOE (Notice of Exception) messages upon completion of benefit update processing.  These messages are transmitted to the appropriate BDNC (Benefit Delivery Network Center).  After further processing, the messages are available for regional office retrieval.  Regional office personnel should obtain NOE messages on a daily basis.  See part II, chapter 8, for procedures for using the NOES (Notice of Exception Subsystem) command to retrieve messages.

**b**.  Each BDNC only retains NOE messages for 30 processing days.  These centers also compare every message against the PIF (Pending Issue File) to determine whether affected pending issues should be cleared or updated (to reflect a reject).

(1)  If a pending issue exists and it is "pending acceptance" and no message is posted for that master record when BDN completes update processing and produces messages, BDN will clear the pending issue.

(2)  If a pending issue exists and it is "pending acceptance" and a message is posted for that master record, BDN will determine whether to clear the pending issue based on the message number.

(i)  If the message is in the 8000-series range, BDN clears the pending issue as these messages do not indicate a rejected transaction

(ii)  If the message is not in the 8000-series range and was generated by award processing, BDN updates but does not clear the pending issue.  The regional office must either cancel the pending issue or clear the pending issue through generation of a corrected transaction.

*NOTE 1:  Some of the messages in this chapter cannot be issued as the result of input from a BDN terminal; however, these messages can be issued as a result of DOD (Department of Defense) changes to a claimant's  Contributions Account.*  
  
*NOTE 2:  Corrections (02V transactions) to some fields in master records can be made directly on a BDN screen.  Seepart II, paragraph 8.13, for allowable entries.*

FOOTNOTES:

1       Messages 2950, 2951, 2952, 2953, and 2960 involve changes to a claimant's Contributions Account.  To properly review these messages, it is necessary to determine which allotment months have been affected, to what extent that the allotment months have changed, and whether an adjustment (i.e., increase or decrease) is necessary.  Maintain a current copy of the M27 screen (and the M28 screen if the claimant has a DOD contribution) in the claimant's folder as a normal part of the chapter 32 claim documentation.  Review these screen prints during routine claims processing.   If the Contributions Account has changed or the existing prints are more than 6 months old, print the appropriate screen(s) to replace the screen(s) in the claimant's folder.   In many instances, the Contributions Account changes which generate the above-mentioned messages will affect all award lines and rates.  In such instances, any amended award action must repeat all award periods.

        When allotments are being made at a low monthly rate (e.g., $25 per month), the allotments can extend beyond a 6-year period.  However, BDN has a 6-year maximum field for allotments.  If additional allotments are received (to exceed the 6-year maximum), BDN posts these allotments to retroactive months.  A retroactive award should not be made in such a case because the new allotments only affect remaining entitlement.  BDN automatically adjusts the award DLP (Date Last Paid) and issues a message.  File the message without action provided that BDN had properly adjusted the award DLP.

2       The transaction code for messages 8024, 8025, and 8026 will be 88G.  This code signifies a system-generated transaction which occurs when a Bank file processing action adjusts a claimant's name or file number (or merges two Contributions Accounts), and the master record also includes a Benefit Account.  The 88G transaction updates the Benefit Account so it reflects the change(s) made during Bank file processing.
