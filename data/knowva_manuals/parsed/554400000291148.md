| **Topic 1: DGIB Managed Service and DGI processing** | |
|  | --- |
| **Introduction** | This topic covers the integration of the DGIB Managed Service and the DGI system for processing claims. |
|  | --- |
| **Change Date** | January 13, 2025 |
|  | --- |
| **II.1.B.1.a**  **General** | As of July 8, 2024, the DGI system integration with the Benefits Delivery Network (BDN) was removed and replaced with the Enterprise Management of Payment Workload and Reporting (eMPWR) system which handles the:    * Transmission of chapter 33 business transactions (BTs) to the Department of Treasury to issue payments,  * Establishment of debts (receivables), and  * Notification of BTs to the Debt Management Center (DMC) for debt management activities. |

|  |  |
| --- | --- |
|  | --- |

|  |  |
| --- | --- |
| **II.1.B.1.b**  **Impacts to Processing Action** | With the transition to eMPWR, there were impacts to processing actions taken by Veterans Claims Examiners (VCEs) before the transition.  These differences include the following actions.    * All manual fiscal transactions (FIST/FAUT actions) needed for the correction or completion of a chapter 33 record must be routed for review and processing by the Regional Processing Office’s (RPO’s) Support Services Division (SSD), unless otherwise indicated.  * The Work Product Summary (WPS) and Payment Instruction (PI) screens of DGI have had significant updates, including authorization options, payment information display, and new Miscellaneous Awards.  * Payment addresses for chapter 33 beneficiaries (including direct deposit and physical addresses for paper checks) are now managed on the Claimant Biography (BIO) screen of DGI.  * Putting a chapter 33 file into suspense or removing such a suspense for accounts with proceeds on file or potential death claims are managed on the BIO screen of DGI.  * No BDN Master Records are created for new chapter 33 beneficiaries, nor are existing master records updated in BDN.  * No BDN Pending Issue Files (PIFs), including diaries and pending end products (EPs), can be created for chapter 33.  * Review of education benefit payments for chapter 33 occur in eMPWR for payments issued after June 28, 2024.  * Notification of payments and debts are managed by one consolidated “Payments Letter” for chapter 33 beneficiaries. |

| **Topic 2: DGIB Managed Service and BM processing** | |
|  | --- |
| **Introduction** | This topic covers the integration of the DGIB Managed Service and the BM system for processing claims. |
|  | --- |
| **Change Date** | January 13, 2025 |
|  | --- |
| **II.1.B.2.a**  **Benefits Manager** | The BM processing system replaces BDN as the primary award processing system for VA Education Service (EDU) for chapter 30, chapter 1606, and chapter 35.   With the initial release of BM on December 7, 2024, beneficiaries applying for chapter 30, chapter 1606, and chapter 35 without any master record (MR) in BDN will be processed using BM as directed by RPO leadership.   With the subsequent migration of BDN records to BM, scheduled for summer 2025, all chapter 30, chapter 1606, and chapter 35 claims will be processed using BM. |

| **II.1.B.2.b**  **Impacts to Processing Action** | With the introduction of BM, there were impacts to processing actions taken by users before the transition.  These differences include the following actions.   * All manual fiscal transactions (FIST/FAUT actions) needed for the correction or completion of a non-chapter 33 record must be routed for review and processing by the RPO’s SSD for any beneficiaries processed in BM.  * Payment addresses for non-chapter 33 beneficiaries (including direct deposit and physical addresses for paper checks) are managed on the Beneficiary Profile screen of BM.  * Putting a non-chapter 33 file processed in BM into suspense or removing such a suspense for accounts with proceeds on file or potential death claims are managed on the Beneficiary Profile screen of BM.  * No BDN Master Records are created for non-chapter 33 beneficiaries processed in BM.  * Before establishing (CEST) any new master record in BDN, the user must search for the beneficiary in BM to ensure no record exists there. |

Date of death.   *Note*: *Call center* users can only modify the payment status if the suspense reason is *Beneficiary payment undeliverable* and the user has validated the payment information.   2. 4. Access and modify the Enrollment Verification screen. 3. *Program support assistant (Work study)*:  This role allows users to take the following actions:    1. Search for beneficiaries.    2. View claims with the user’s sensitivity level.    3. Modify all fields on the Beneficiary information screen except:       1. File number,       2. POA,       3. Six-credit hour exclusion,       4. Date of birth, and       5. Date of death.    4. Access and modify the Enrollment Verification screen.    5. Create a new claim and add or edit a *Work study* manual award on the Enrollments screen. 4. 6. Access the other claim screens in read-only format*.*    7. Authorize a claim if only *Work study* awards display in the Net awards section of the Claim summary screen. 5. *Trainee*:  This role allows users to take the following actions:    1. Search for beneficiaries.    2. View claims with the user’s sensitivity level.    3. Create, modify, and update claims.    4. Delete information within a claim and an in-progress claim.    5. Access and modify the Enrollment Verification screen.    6. Adjudicate and authorize claims. |

| **II.1.B.2.d**  **Traffic Marshal** | ***NOTE*: Traffic Marshal functionality and reporting is expected to be turned off upon the successful migration of BDN award data post the sunset process, allowing for processing in BM.**  A *Traffic Marshal Fail Safe Report* is available on the Power BI dashboard that includes any beneficiary who has a file in both BDN and BM.  The report includes the following information:   1. Station, 2. Beneficiary SSN, 3. Veteran VA file number, 4. Date added to BM, 5. Date added to BDN, and 6. Benefit type.   This report is intended to be a final protection against locate beneficiaries being erroneously processed in both BDN and BM so that the RPO of jurisdiction can take the appropriate remediation action.  Procedural and systematic controls are in place to limit such an occurrence.   1. BM has a built-in system check, known as the Traffic Marshal, that prevents users from creating claims in BM for any beneficiaries who have any existing master record in BDN.  2. Any time a new beneficiary is created in BDN, users must confirm that no such beneficiary has previously been created in BM.  Failure to do so could cause duplicate records to be created in BM and BDN which can lead to duplicate payments.   For any beneficiary that appears on the *Traffic Marshal Fail Safe Report*, the RPO of jurisdiction must take appropriate action to correct any duplicative actions and to ensure that no additional duplication occurs.  In these cases, users must CORR a STOP message, “STOP, process in BM” on the M21 screen in BDN and add a Flash message to the Image Management System (TIMS) file.  Claims will be processed in BM.  If BM lacks the functionality to process a specific beneficiary’s claim or BM erroneously allows the creation of a claim for a beneficiary with a BDN master record, follow local procedures to submit a Service Now (SNOW) ticket requesting the DGI Production Operations (ProdOps) team remove the beneficiary profile from BM.    If duplicate payments were erroneously issued, users must take appropriate action to create the appropriate debt(s) in BDN and to notify the beneficiary of the incorrect payment. |
