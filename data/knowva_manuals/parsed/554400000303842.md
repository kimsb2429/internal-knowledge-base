| **II.3.J.1.a**  **General** | Benefits Manager contains functionality to process a claim for a chapter 30 death refund, submitted by the next-of-kin of a deceased Veteran.  For more information on the chapter 30 death benefit, including eligibility criteria, see M22-4, Part V, 4.11. |
|  | --- |

|  |  |
| --- | --- |
| **II.3.J.1.b**  **Claim creation** | To create a chapter 30 death refund claim, a user will navigate to the Claim history tab of the deceased Veteran and select the “Create a claim” button. In the Create a claim modal, select the “Chapter 30 (MGIB-AD) - Death refund” option.  **Note:** In order for the “Chapter 30 (MGIB-AD) - Death refund” claim option to appear, the Veteran’s bio page must contain a date of death. |
|  | --- |

|  |  |
| --- | --- |
| **II.3.J.1.c**  **Claim usage** | A chapter 30 death refund may be used to process the following scenarios:   * Development to next-of-kin for claim after receipt of notification of death * Award of benefits to next-of-kin * Denial of claim when Veteran is deceased but no entitlement to death benefit exists |
|  | --- |

|  |  |
| --- | --- |
| **II.3.J.1.d**  **Eligibility section** | The Eligibility section of chapter 30 death refund claim contains reduced functionality in comparison to the Eligibility section of an original or supplemental chapter 30 claim.  The “DoD participation code & contributions” section contains chapter 30 participation data. As in a normal chapter 30 claim, the user may override this data with the “Add VA participation override” button, if necessary. For more information on the VA participation override functionality, see M22-4, Part II, 3.D.3.d.  The user may also add additional contribution (also referred to as buy-up) data by using the “Add additional contribution” button. For more information regarding input of additional contributions, see M22-4, Part II, 3.D.3.e**.**  The “Chapter 30 (MGIB-AD) service period(s)” section allows for input of service data by using the “Add service period” and “Add reserve service obligation” buttons. Service data is not considered during death refund processing and is included for informational purposes only. For more information regarding input of service, see M22-4, Part II, 3.D.3.f**.**  The “Chapter 30 death refund” section contains the “Refundable contributions” field. The amount in this field is calculated by system rules, based on user inputs and previously awarded benefits. If the Veteran was paid benefits in Benefits Manager prior to their death under chapter 30, the total amount paid will reduce the refundable contributions. If the amount paid exceeds or is equal to the total basic and additional contributions, the field will display $0.00. |
|  | --- |

|  |  |
| --- | --- |
| **II.3.J.1.e**  **Next of kin information section** | The Next of kin information section of chapter 30 death refund claim contains details on the next-of-kin who are eligible for the death benefit or are potentially interested parties.  This section displays the Refundable contributions amount discussed in M22-4, Part II, 3.J.1.d**.** It is included to aid the user in determining the appropriate refund amount for each next-of-kin.  To add a next-of-kin, the user will select the “Add next of kin” button.  Within the “Add next of kin” modal, the user will be presented with the following fields:   * Relationship type   + This is a required field and must be populated.   + Acceptable values include:     - Father     - Mother     - Spouse     - Child     - Other * First name   + This is a required field and must be populated. * Middle name * Last name   + This is a required field and must be populated. * Claimant ID   + Currently, only the Social Security of the next of kin should be entered in this field   + If the next-of-kin does not have a Social Security Number, or an SSN is not of record, the user should select the “There is no Social Security Number” checkbox. This selection must be further affirmed by checking the “I verify there is no Social Security Number associated with this claimant.” checkbox. * Date of birth   + This is a required field and must be populated. * Mailing address   + The functionality in the subsidiary fields of this section are documented in M22-4, Part II, 3.C.1.d. * Refund amount   + This field may be left blank if the next-of-kin are being established prior to receipt of claim.   + The amount input cannot exceed the value of the “Refundable contributions” field   + Upon save of the modal, the total sum of the refund amounts of all next-of-kin is validated against the refundable contributions amount. If the total sum exceeds the refundable contributions, an error stating, "**The sum of the next of kin refund amounts cannot exceed the refundable contributions"**, will display.   + In order to resolve, the refund amounts amongst all next-of-kin must be adjusted so that the total sum is less than refundable contributions.   **NOTE:** It is recommended that all next-of-kin have a Beneficiary information page established prior to being added to the Next of kin information page. See M22-4, Part II, 3.C.1.bfor more information on establishing a Beneficiary information page. |
|  | --- |

|  |  |
| --- | --- |
| **II.3.J.1.f**  **Claim summary section** | The chapter 30 death refund claim uses the same Claim summary workflow as an original or supplemental chapter 30 claim. For more information on the Claim summary section of a claim, see M22-4, Part II, 3.G**.**  Two signatures are required for all chapter 30 death benefit claims which contain awards. If the claim contains no awards, a user may submit for authorization by using the “Submit for authorization” button or may authorize the claim by using the “Authorize” button. Signature requirements for chapter 30 death refund claims will be dictated by RPO management. |

|  |  |
| --- | --- |
| **II.3.J.1.g**  **Letters generated by chapter 30 death refund claims** | The chapter 30 death refund claim does not produce a system-generated letter. Users may upload an appropriately information user-generated letter using the procedures detailed in M22-4, Part II, 3.H.1.c. |

| **II.3.J.2.a**  **General** | Benefits Manager contains reduced functionality necessary to process a chapter 30 Transfer of Entitlement (TOE) claim.  For more information on chapter 30 Transfer of Entitlement, see [Circular 22-02-02](https://vaww.vrm.km.va.gov/system/templates/selfservice/va_kanew/help/agent/locale/en-US/portal/554400000001048/content/554400000058825/22-02-02-Transfer-of-Entitlement?query=%2222-02-02%22). |

| **II.3.J.2.b**  **Entitlement adjustments for transferring Veteran** | If a Veteran is approved for chapter 30 TOE, their chapter 30 entitlement must be adjusted to reflect the amount of entitlement transferred out to their dependents.  If the Veteran’s record does not already reflect their chapter 30 TOE transfers, a user must create and authorize an original or supplemental claim for the Veteran and adjust entitlement as necessary using the manual entitlement adjustments functionality detailed in M22-4, Part II, 3.E.2.b. |
|  | --- |

|  |  |
| --- | --- |
| **II.3.J.2.c**  **Claim creation for beneficiary** | To create a chapter 30 TOE claim, a user will navigate to the Claim history tab of the dependent beneficiary and select the “Create a claim” button. In the Create a claim modal, select the “Chapter 30 TOE (MGIB-AD)” option and click the “Continue and establish relationship” button to proceed to the next step. |
|  | --- |

|  |  |
| --- | --- |
| **II.3.J.2.d**  **Establishing Veteran-beneficiary relationship** | After clicking the “Continue and establish relationship” button, the user will be presented with the “Establish relationship” modal. This modal contains a single field, Relationship type. This field is required and has the following acceptable values:   1. Spouse 2. Child   After selecting a relationship type, the user will click “Continue”. They will be presented with an additional filed, “Veteran information”. This field is required and acceptable inputs are the Social Security Number or file number of the transferor Veteran. This information is used to locate and create (if necessary) a Veteran in the Claimant Service. For more information on searching for a Veteran during relationship establishment, see M22-4, Part II, 3.C.3.c. |
|  | --- |

|  |  |
| --- | --- |
| **II.3.J.2.e**  **Eligibility section** | The Eligibility section of a chapter 30 TOE claim contains minimal information required to establish eligibility.  The “Relationship information” table will display the relationship type and transferor Veteran’s name and file number. This information is read-only and for informational purposes only.  The “Delimiting date” table is conditionally required. If the beneficiary is a spouse, a delimiting date is not required but may be input if appropriate. If the beneficiary is a child, a delimiting date is required to be entered by the user. If a delimiting date is appropriate, the user will click the “Add delimiting date” button. The user will be presented with the “Add delimiting date” modal, where the user can enter the appropriate delimiting date or use the date picker by clicking the calendar icon.  If a delimiting date has been previously entered, a pencil icon will appear next to the delimiting date. If a user clicks the pencil icon, they will be taken to the “Edit delimiting date” modal. The functionality of this modal is identical to the “Add delimiting date” modal. Additionally, if the relationship type is a spouse, a trashcan will appear, which will allow for deletion of the delimiting date if appropriate. |
|  | --- |

|  |  |
| --- | --- |
| **II.3.J.2.f**  **Entitlement section** | The Entitlement section of a chapter 30 TOE claim contains minimal information required to establish entitlement.  The “Manual entitlement adjustments” section will allow for initial establishment and adjustment of entitlement. If a beneficiary is eligible, the user will click the “Adjust manual entitlement” button. The user will be presented with the “Adjust manual entitlement” modal. For more information on this modal, see M22-4, Part II, 3.E.2.b. The user will add the appropriate amount of entitlement which has been transferred to the beneficiary and click the “Submit adjustment” button.  If necessary, previously saved adjustments may be edited or deleted using the pencil and trash can icons, respectively, as detailed in M22-4, Part II, 3.E.2.b. |
|  | --- |
| **II.3.J.2.g**  **Enrollments section** | The Enrollments section of a chapter 30 TOE claim contains minimal information required to establish awards.  The “Manual awards” section will allow for input of the following award types which are present in a normal chapter 30 workflow:   * Accelerated payment * Apprenticeship * Correspondence * Flight * Licensing and certification * National exam * On-the-Job training * Tutorial assistance * Work study   For more information on the input of these types of awards, see M22-4, Part II, 3.F.7.  Four additional manual award types are also available in the chapter 30 TOE workflow:   * Graduate * High school * NCD * Undergraduate   Upon selection of one of these manual awards, the user will be presented with the fields detailed in M22-4, Part II, 3.F.2.a-c, as applicable. These entries are for informational purposes only and are not used to calculated awards or entitlement.  There is an additional field, “Entitlement charge”, which only appears within the chapter 30 TOE workflow. This field is required and determines the entitlement charge for the period which is used by the system rules. The user should input the appropriate entitlement charge for the payable period in MM-DD format.  A user can add payment for a manual award period by clicking the down arrow on the left of the award and selecting the “Add payment” button. The user will be presented with the “Add payment” modal. In this modal, the user will establish monthly payments for each stub or whole month within the enrollment period. The following fields are available:   * Begin date   + This is a required field and should be the beginning of the enrollment, or the beginning of a month within the enrollment, as appropriate. * End date   + This is a required field and should be the end date of the enrollment, or the end date of the month used for the begin date, as appropriate. * Amount   + This is a required field and should be the total amount due for the time period established by the Begin and End date fields. * Description   + This field is optional and can be up to 200 characters.   After establishment of a payment, the pencil and trash can icons will appear for each payment. These icons can be used to edit or delete the payment entry, respectively. |
